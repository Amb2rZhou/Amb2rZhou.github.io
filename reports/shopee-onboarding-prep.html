一、角色校准

你不是 search PM，也不是 chatbot RD。你的角色更准确叫"对话式商品策略 PM" —— 你定义的是"用户用自然语言时，Shopee 怎么搜+推+卖"，搜推算法和 LLM 都只是你工具箱里的一部分。

这个定位决定了你的能力树和 RD/算法 PM 不一样：

  • 不需要会写 transformer，但得能在白板上跟搜推算法、LLM RD、商品中台、客服团队同时对话且不掉线
  • 不需要训模型，但得会算 token 成本、定 latency budget、设计评测体系
  • 核心交付物：PRD + 数据分析报告 + AB test 设计 + 跨团队协调
一面情报回顾：Shopee 搜索 AI 三层职能 = ① 站内 toC 导购 + ② 内部员工 Copilot + ③ ChatGPT MCP。你被分到 ① 的概率最高，但跨方向轮岗也可能。下面所有内容都按 ① 写，但底层框架对 ②③ 都通用。

二、能力树（按 ROI 排，不是按字典序）

🔥 Tier 1 · 进去 1 个月内必须能讲清的

1. 电商搜推三板斧 + 它们和 chatbot 的关系
  • 召回（Recall）：传统倒排 / 向量召回 / 多路融合 → chatbot 里 LLM 改写 query 后还是要走召回，chatbot 不替代召回，而是改它的输入
  • 排序（Ranking）：粗排 → 精排 → 重排，电商和通用搜索的差异是精排里要塞商业目标（GMV / 加购率 / 利润率 / 商家关系）
  • 相关性 vs 商业 vs 个性化：三者永远在打架，PM 的活就是定义"在 chatbot 这个场景下怎么权衡"
2. AI 导购 chatbot 的对话漏斗（你之后所有 metric 设计都从这里来）
站内 PV
  ↓ chatbot 触发率（用户主动开口的比例）
触发对话
  ↓ 多轮持续率（第一句之后还聊 ≥ 2 轮）
深度对话
  ↓ 转 SKU 率（聊出具体商品）
具体商品
  ↓ 加购率
加购
  ↓ 下单转化率
GMV

每一层对应一组 metric + 一类设计问题。PM 入职就该明确自己负责哪一层——不是全要，而是 own 1-2 层做透。

3. AI 的能 vs 不能
AI 擅长AI 不擅长 / 别交给它
跨品类联想（"送男朋友礼物"）精确库存 / 物流时效（让传统系统答）
模糊意图收敛价格敏感场景（用户只看降价标签）
自然语言对比说明刚需明确商品（用户想买"小米 14"，别让他多聊）
个性化解释 / 推荐理由售后纠纷（合规/法务红线）

好 PM 的设计直觉 = 知道什么场景该让 AI 上、什么场景该 graceful 降级回传统流程。

4. SQL + 看 metric + 读 case 的能力
  • SQL 不是炫技，是让你能自己回答问题而不是每次找数分排队
  • 读 chatbot log → 抽 100 条对话样本看 → 看出"用户为什么放弃"是 PM 最值钱的肌肉
  • AB test：双边显著性 / 长短期 metric 冲突 / 实验污染 / pre-AA 检验

📚 Tier 2 · 3 个月内补齐

5. Prompt engineering（PM 视角，不是 RD 视角）
  • 写 prompt 不是炫技，是定义产品行为的最小颗粒
  • PM 应该懂的：system prompt vs few-shot 的边界 / RAG context 注入策略 / hallucination 在你的产品里长什么样
  • 重点：能跟 LLM RD 在 prompt 设计上「吵起来」，不是只能说"再调一调"
6. RAG over 商品库
  • 商品 embedding 怎么选（CLIP / 文本 / 多模态）
  • 召回 → 截断 → 重排 → 注入 prompt 的 budget 怎么分
  • 商品属性结构化字段 vs 自然语言描述 在 RAG 里各自怎么用
7. Tool calling 设计
  • chatbot 调内部 API（库存 / 价格 / 优惠券 / 加购）的 contract
  • 失败降级 / 部分成功 / 异步 / 并发
  • 能让 chatbot "操作"而不止"推荐"是产品天花板的差距（加购、领券、设到货提醒、跨店凑单都是 tool）
8. Persona / 风格设计
  • "温暖店员" / "专业顾问" / "闺蜜推荐" / "讨价还价的小贩" —— 东南亚 7 个市场可能要多套
  • chatbot 是 Shopee 品牌的延伸，Tone 错了用户对整个品牌印象都会变

🌱 Tier 3 · 长期慢慢长

9. 跨团队协作
  • 搜推算法（核心搭子）/ 商品中台（数据上游）/ 商家服务（商家利益）/ 客服（边界划分）/ 法务（合规）
  • 每个团队听到"AI 导购"想到的东西不一样——PM 的活之一是翻译
10. 东南亚电商特有
  • 7 国 6+ 语言，不是翻译问题，是 cultural persona 问题（印尼用户和越南用户讨价还价方式都不一样）
  • 节庆节奏（5.5 / 9.9 / 11.11 / Ramadan / Tet）
  • COD 占比高 → 用户决策链路 / 信任成本和淘宝京东不一样

三、5 个核心设计框架

框架 1 · 场景驱动 > Feature 驱动

不要问"chatbot 应该有哪些功能"，问"用户在什么场景下，自然语言比现有 UI 更好用"。

高 ROI 场景为什么
送礼推荐模糊意图 + 用户不愿筛选 + 跨品类联想
补货 / 复购用户已知道想要什么类，但懒得每次重新找
跨品类比价传统搜索没法跨品类对比
长尾品类教育用户不知道用什么关键词搜
节日大促决策信息过载，需要"帮我决定"
低 ROI / 别做为什么
纯关键词导航传统搜索框完胜
已知 SKU 购买直接搜更快
严肃售后投诉合规风险 + 用户已经在气头上

PM 的功夫 = 找到 chatbot 真正不可替代的场景，而不是把所有场景都塞进去。

框架 2 · 信任堆叠模型

AI 导购的核心障碍不是技术，是用户为什么相信 chatbot 的推荐。信任 = 一系列设计选择堆出来的：

准确性     → 推荐确实跟需求匹配（基础）
        +
多样性     → 不只推贵的 / 不只推一家店
        +
透明度     → 每条推荐能解释清楚理由
        +
退出权     → 用户随时能切回传统搜索
        +
一致性     → 同一个问题第二次问，答案还稳
        +
责任承担   → 推错了怎么补偿（这是商业问题不是技术问题）

每一层都是设计问题。第一版 chatbot 上线时至少要前 4 层，否则用户用 1 次就再不来了。

框架 3 · 对话状态机（不是开放对话！）

PM 圈一个常见误区：「让用户随便聊」。电商 chatbot 不能这么设计——开放对话 = 失控 = 转化率塌方。

实际有效的设计是 半开放状态机：

[Greeting]
   ↓
[Discovery]      ← 用户说"我想买...", AI 提取意图、品类、价格、场合
   ↓
[Clarification]  ← AI 反问 1-2 个关键属性（不超过 2 轮，不然用户烦）
   ↓
[Recommendation] ← AI 给 3-5 个具体商品 + 推荐理由
   ↓
[Objection]      ← 用户说"太贵了 / 我想要红色的 / 还有别的吗"
   ↓
[Close]          ← AI 引导加购 / 领券 / 设提醒

每个状态有进入条件 + 退出条件 + 兜底分支。这套设计你以后会画 N 遍——它就是电商 chatbot 的 wireframe。

框架 4 · Tool 设计 = 产品天花板

最容易被忽视的设计层：chatbot 能调哪些 tool，决定了它的产品上限。

Tool 等级能力用户感受
L1 只读查商品 / 查价格 / 查评价像个会说话的搜索框
L2 操作加购 / 领券 / 设到货提醒像个真助理
L3 协商跨店凑单 / 拼团 / 砍价像个会讨价还价的导购
L4 跨域调物流 / 售后 / 退换像个全能管家
Shopee 现状大概在 L1，你能推到 L2 就值得转正。L3 是真正的差异化（vs Lazada / Amazon Rufus），但需要跨团队协作，要等。
框架 5 · Cost × Latency × Quality 三角

PM 不会算 token 费用是大问题。粗略数量级感受：

  • 一次多轮对话 ≈ 5-10K tokens
  • 用 GPT-4 / Claude Sonnet：~$0.05-0.15 / 对话
  • 用开源模型自托管：~$0.005-0.02 / 对话
  • Shopee 一天 toC 对话量如果跑到 100 万 → 每天成本 5-15 万 RMB vs 5000-2 万 RMB，年化差距巨大

PM 要参与的决策：

  • 主对话用什么模型 / 子任务（query 改写、属性抽取）用更小的模型
  • Latency budget：用户愿意等多久（电商场景 P95 < 2.5s 比较合理，>3s 流失明显）
  • 怎么 cache / 怎么压缩长 context

这是 PM 跟算法/工程最常打架的地方，提前心里有数。

七、入职第一周 4 个具体动作

不是"学知识"，是用动作把自己嵌进团队：

  1. 跑通现有产品：把 Shopee 7 个市场 app 都装上，找出现有所有 AI 触点（搜索建议、商品页问答、客服 bot、购物车凑单），抽 50 条对话样本写成笔记
  2. 画一张 chatbot stakeholder 图：搜推算法 / LLM RD / 商品中台 / 商家服务 / 客服 / 法务 / 业务运营，每方关心什么、利益冲突在哪
  3. 找出现有 chatbot 漏斗的最大漏点：哪一层流失最严重，为什么。这是 PM 的「破冰项目」
  4. 拉一份 Lazada / Amazon Rufus 的最新动态：用 AFI 现成的 daily brief 抓取能力扫一遍他们最近 3 个月发布会/产品更新
每件事都能产出一份给 mentor 看的 deliverable，第一个月转正预期就稳了一半。

九、单独提醒的风险点

风险 1：chatbot 不是过气，是在变形（2026-05 update）

原版风险点写的是"chatbot 可能过气"——基于 2024 年判断。2026-05 数据已经反过来：chatbot 不止没过气，是在升级成 agent。

看这一年的关键节点：

  • Amazon Rufus 用户 +115% MAU / +400% interactions YoY，2025 年贡献 $12B 增量销售 —— 用户用得越来越多
  • Rufus 4/28 上线 Scheduled Actions —— chatbot 已经能自主下单，从"对话助手"变成"自动化购物 agent"
  • 京东京小智 5.0 多 Agent 协作已经验证 ROI（-28% 转人工 / +37% 转化）—— 不再是实验
  • 阿里千问把 AI 助手做成横跨业务线的统一入口 —— chatbot 是一个入口，但同一个 AI 在所有 app 里

真正的风险已经变成：

  • 风险 1a：你做的"chatbot"半年后会不会被要求升级成 agent？（很可能会）
  • 风险 1b：chatbot 入口会不会被合并进搜索框？（Amazon 在评估这个）
  • 风险 1c：toC 流量主战场是不是其实在视频/直播侧？（TikTok 走的就是这条 —— 入职后第一周就该问到 Shopee 流量构成数据）

对策：

  • 你的能力树要从"chatbot PM"提前升级到 "agentic commerce PM" —— 多看 tool calling / scheduled actions / autonomous agents 这条线
  • 对 mentor 说 "我想从 L1 只读 chatbot 推到 L2 操作型" —— 这跟行业方向是同步的，不是激进
  • 留意 Shopee 内部对"chatbot vs 搜索框融合"的讨论 —— 这个决策影响你产品的存在感

风险 2：转正率 20% × 时间冲突 = 实际转正率可能更低

Shopee 的转正逻辑大概率是"实习期能完整跨越的人优先"。你 6 月底走、7 月去蚂蚁，对 Shopee 而言你是过渡型实习。这意味着：

  • 转正概率客观上低于 20%（除非你做得极好）
  • 但 Shopee 履历 + AI 导购 chatbot 实战经验本身就有价值，不一定要奔着转正去
  • 战略选择：把 Shopee 当作"AI PM 实战学校"，目标是积累简历 + 行业 context，而不是死磕转正

风险 3：搜推 vs LLM 两边的政治

电商公司的搜推团队往往是老牌强势团队（绩效核心）。LLM/AI 团队是新势力（流量 + KPI 还在搭建）。

chatbot 这种产品天然处于两边交叉地带，PM 容易夹在中间。第一周就观察清楚团队权力结构：你 mentor 的位置 / 搜推总监 vs LLM 负责人的关系 / 谁拍板。

风险 4：东南亚多语言不是翻译问题

很多新人会以为"翻译就行"。实际：

  • 越南语 / 泰语 / 印尼语 LLM 模型质量明显低于英文
  • 每个市场的购物文化 / 节庆 / 价格敏感度都不同
  • 同一个 prompt 在 7 个市场可能要 7 个版本

对策：第一周就跟某个非英文市场的运营聊一次，听他们 complain 现有 AI 怎么不本地化。
