6b.1 大模型语料采集 6 题（蚂蚁风格的 AI 专业题）

Q. 大模型语料采集和清洗应该注意什么？
  • 采集维度：多样性（避免偏差）/ 时效性 / 合规性（版权 / 隐私）/ 质量分层
  • 清洗维度：去重（fuzzy match）/ 噪音过滤（乱码 / 广告 / 低质 UGC）/ 格式标准化 / 毒性/违规内容过滤
  • 金融场景特殊：PII 脱敏（身份证 / 账号 / 金额）+ 合规关键词过滤 + 引用溯源
Q. 如何评判大模型语料采集的质量？

分四维度：

  1. 完整性：是否覆盖 target domain 的关键子域（可用 topic modeling 检测 gap）
  2. 准确性：事实一致性（人工 spot check + LLM-as-judge 批量评估）
  3. 多样性：语言风格 / 观点立场 / 时间跨度
  4. 时效性：数据新鲜度分布

举具体例子（如果面试官追问）：金融语料评判——"好"的金融语料应该覆盖研报 / 新闻 / 公告 / 行业分析 / 监管文件等 5 类来源，缺任何一类都有 gap。

Q. 如何计算大模型语料采集的完整性？
  • 先定义领域分类 schema：比如金融分 18 个子域（权益 / 固收 / 量化 / 合规...）
  • 采样统计：每个子域的样本占比 vs 期望占比（用 chi-square 检测分布偏差）
  • embedding 空间覆盖率：把语料和"理想全集"都 embed，看 embedding cluster 覆盖率
  • 外部验证：用 held-out 问题集（未见过的 query），看模型能不能答对 → 答不对的说明语料覆盖不够
Q. 大模型语料采集的难点是什么？
  • 高质量语料获取成本高：专业内容（法律/金融/医疗）都有版权，爬虫违规风险
  • 合规和可用性的平衡：脱敏后语料的语义损失 vs 合规刚性要求
  • 长尾场景难覆盖：通用语料好找，专业细分场景（如"东南亚伊斯兰金融合规"）找不到
  • 质量评估主观性：没有 ground truth，评估依赖人工或 LLM-as-judge，成本高
  • 语料时效 decay：金融语料半年前可能已过时，需要持续更新但成本高

6b.2 Boss 的 AFI 挂钩点

这些大模型语料题你可以用 AFI 的经验回答——AFI 本质就是做多源信号采集 + LLM 相关性打分：
  • 采集：RSS / GitHub / ArXiv / HuggingFace 多源（vs 大模型多源）
  • 打分：LLM 按相关性筛选（vs 大模型质量评估）
  • 去重：我处理过 signal 的去重（vs 大模型语料去重）
  • 时效：AFI 的 daily 机制（vs 语料更新机制）
"我做 AFI 的时候在小体量上踩过这些坑，对大模型语料采集的难点有直观体感。"

Layer 1：MaaS 中台（对应蚂蚁百灵）

  1. 多语言配置模板：让各 BU 搭 AI 应用时，语言 / 合规模板一键切换，不用每国重新开发。

Layer 4：toC 产品（Alipay+ · 跨境汇款 · 理财顾问 · AI 客服）

Q2: 蚂蚁核心业务 + AI 在哪些环节创造价值

蚂蚁核心是四块业务：

1. 支付（支付宝） - AI 价值：实时风控反欺诈。每秒处理海量交易，AI 识别异常模式（异地登录、非常规金额、设备指纹异常），毫秒级拦截。这是 AI 最成熟的金融应用。

2. 数字金融（理财/信贷） - AI 价值：个性化投顾。不是按收益率排列表，而是根据用户的风险承受能力、资金使用计划、家庭结构给出个性化方案。把"专业投资顾问"的能力普惠化。

3. 信用（芝麻信用/花呗/借呗） - AI 价值：更精准的信用评估。传统征信只看收入和还款记录，AI 可以加入行为数据（消费稳定性、社交图谱、还款主动性），覆盖传统征信覆盖不到的人群。

4. 保险（蚂蚁保） - AI 价值：智能核保（降本）+ 个性化推荐（增收）+ 理赔辅助（提升体验）。今天我体验了蚂蚁保的 AI 推荐，它会预填年龄、常住地、收入——说明已经在用支付宝的数据做个性化了。

贯穿四条线的最大 AI 价值：蚂蚁拥有用户最完整的金融 Context。 收入看得到（工资流水）、支出看得到（花呗/账单）、信用看得到（芝麻分）、资产看得到（余额宝/基金持仓）。其他任何平台只有片段，蚂蚁有全景。这是做 AI 金融产品最强的原料。

Q3: 智能理财助手怎么设计？核心功能和差异化

思考框架：用户旅程拆解——了解需求→给建议→持续服务。

我会分三个阶段设计：

阶段一：了解用户（需求采集） - 不让用户填风险测评问卷——大家都乱填，数据不真实 - 改为对话式采集：问"你这笔钱什么时候用？"、"能接受最大亏多少？"、"有没有房贷在还？" - 几轮对话比一张问卷精准得多，而且用户感受是"有人在关心我"而不是"在填表" - 蚂蚁的独特优势：很多信息不用问——收入看工资流水就知道了，支出看账单就知道了，AI 只需要确认和补充，不需要从零收集

阶段二：给建议（推荐+解释） - 不推 20 个产品让用户自己选。推 2-3 个，每个附上推荐理由 - 比如："推荐这款短期理财，因为你说了半年后要用钱，它 6 个月到期，年化 3.2%，风险等级低" - 关键是说清楚为什么，不是给一个排行榜 - 如果用户说"太保守了"，AI 要能调整策略再推一版

阶段三：持续跟踪（动态陪伴） - 买了不是结束，是开始 - 市场大跌时主动推送："你的持仓跌了 5%，但按你的投资计划这是正常波动，不建议卖。历史上类似跌幅 3 个月内回本的概率是 78%" - 产品到期前提醒："你的 XX 理财下周到期，要续还是取出？" - 这是传统推荐做不到的——传统推荐是一次性的，Agent 是持续关系

差异化总结： 别的平台做理财推荐，用户要从零告诉 AI 自己的情况。蚂蚁不用——你的收入、支出、信用、资产它已经知道了。这不是技术差异，是数据壁垒。

Q4: 创新 vs 合规怎么平衡

思考框架：分层处理——硬约束用规则，软体验用 AI。

很多人觉得合规和创新是矛盾的，我不这么看。合规是地板不是天花板——在地板之上可以自由创新。

具体分三层：

第一层：硬合规用规则引擎（绝对不能触碰的底线） - 适当性管理：不能给不合格投资者推高风险产品——这个规则硬编码在系统里，AI 绕不过去，prompt 再怎么写也不可能违反 - 风险提示：法规要求的提示语必须展示，不是 AI 决定要不要说，是必须说 - 数据隐私：哪些用户数据可以用于推荐、哪些不行，在数据层就做好隔离

第二层：软体验用 AI 优化（在合规框架内做到最好） - 怎么用用户听得懂的话解释风险——同样是风险提示，"本产品风险等级 R3"和"这个产品可能亏损本金的 10-20%，适合能承受短期波动的人"效果完全不同 - 什么时机推什么产品——用户刚发了工资 vs 刚还了房贷，推荐策略应该不同

第三层：建立合规审计追踪 - AI 的每一次推荐决策都记录日志：推了什么、为什么推、用户的风险等级是多少 - 这既是监管要求，也是产品迭代的数据源——回头看哪些推荐转化了、哪些被拒绝了

Q5: AI 怎么帮用户理解和选择保险

思考框架：用户痛点拆解——看不懂、选不了、理赔难。

保险最大的痛点是用户看不懂。一份保险合同几十页，保什么不保什么、免赔额怎么算、等待期多久，普通用户根本搞不清。AI 可以在三个环节帮忙：

1. 用人话解释条款（解决"看不懂"） - 用户问"我骑电动车摔了这个意外险赔不赔"，AI 直接回答，不用翻 20 页合同 - 用户问"免赔额 1 万是什么意思"，AI 举例："就是你自己先出 1 万，超过的部分保险公司才管" - 核心是把 B 端的专业术语翻译成 C 端的生活语言

2. 场景化推荐（解决"选不了"） - 不问"你要什么保险"——用户不知道自己需要什么 - 改为问生活场景："你家几口人？有没有小孩？开不开车？有没有房贷？" - 然后推一个组合方案："建议医疗险+意外险+定期寿险，总共每月 XX 元" - 蚂蚁的优势：很多信息支付宝已经知道了。今天我体验蚂蚁保的 AI 推荐，年龄、常住地、收入都是预填的——不需要用户重新输入

3. 理赔辅助（解决"理赔难"） - 出险时用户最慌。AI 可以做：拍照上传 → 预判能不能赔 → 告诉你需要什么材料 → 追踪理赔进度 - 把理赔从"跑三趟柜台"变成"手机上搞定" - 这是降低保险"买了用不上、用的时候很麻烦"的核心痛点

Q6: 金融 AI 最大的风险？怎么控制

思考框架：按风险类型分层，每层对应不同控制手段。

三个层面的风险，严重程度递增：

1. 合规风险（法律层面） - 金融产品推荐有法规要求（适当性管理），AI 不能给不合格投资者推高风险产品 - 控制手段：在 Agent 里硬编码合规规则，不是靠 prompt 软约束。Prompt 可以被绕过，规则引擎不会

2. 误导风险（信任层面） - AI 说"这个产品收益不错"，用户可能理解为"保证赚钱" - 控制手段：金融场景的措辞必须经过合规审核模板。每句话都要区分"事实陈述"和"投资建议" - 反面案例：如果 AI 产生幻觉说了一个不存在的收益率，用户据此投资亏了钱，这是重大事故

3. 操作风险（资金层面） - Agent 如果能直接帮用户买卖，一旦出 bug 或被攻击，影响是真金白银 - 控制手段：三级权限设计 - 低风险：查余额、看持仓、查费率 → 全自动 - 中风险：买理财、配保险 → Agent 给方案，用户确认才执行 - 高风险：大额转账、贷款申请 → 必须跳转原有流程+二次验证

核心原则：金融 Agent 的角色是顾问，不是操盘手。 它可以分析、建议、提醒，但最终下单的按钮必须由用户自己按。这跟我做 intern-clawd 的设计原则一致——"宁可多问一步，不可自作主张"。

Q7: B 端 vs C 端 AI PM 区别？你偏向哪个

思考框架：从决策者、核心指标、容错度三个维度对比。

最大的区别不是产品形态，是谁做决策、怎么衡量成功：

1. 决策者不同 - C 端：用户自己决定用不用。产品要直觉好用，用户没耐心看说明书 - B 端：采购决策和使用者分离。老板决定买不买（要看 ROI），员工决定用不用（要好用）。PM 要同时说服两个人

2. 核心指标不同 - C 端：DAU、留存、转化率、NPS - B 端：客户续约率、使用深度、降本金额、ROI

3. 容错度不同 - C 端可以容忍一定的不准确——AI 推荐错了用户划走就行 - B 端对准确率要求极高——AI 风控漏了一笔欺诈交易就是真金白银的损失；AI 客服回答错了一个保险条款可能引发法律纠纷

我的经历两边都有： - 情报系统偏 B 端思维——服务高管团队，准确率和可信度是第一位的。输出要有来源、要经过验证 - intern-clawd 偏 C 端思维——开源面向个人用户，上手门槛和体验是第一位的。三分钟装好，装了不满意一键卸载

如果要选，我更倾向 B 端 AI。因为我的经济学背景让我擅长算清楚 ROI（"这个 AI 客服能帮你每天省多少钱"），技术背景让我跟工程师沟通顺畅。这两个能力在 B 端比 C 端更值钱。

Q8: 医疗 AI 幻觉怎么防

思考框架：从产品机制设计角度，建四层防护。

医疗 AI 幻觉的后果是：用户按照错误建议吃了不该吃的药，可能出人命。所以产品设计的核心原则是永远不让 AI 成为唯一的信息来源。

第一层：来源溯源 - AI 给出任何健康建议，必须附上出处："根据 XX 临床指南第 X 条" - 用户能点进去看原文 - 没有出处的话 AI 不允许给建议，直接说"我不确定，建议咨询医生" - 这跟我做情报系统的溯源规则一致——每条 signal 必须追溯到原始来源

第二层：分级响应 - 低风险：AI 直接回答（"感冒多喝水多休息"） - 中风险：AI 给建议但加 disclaimer（"以下仅供参考，请以医嘱为准"） - 高风险：直接拒绝回答并引导就医（"你描述的症状建议立刻去急诊"） - 分级标准可以用规则引擎 + LLM 双重判断

第三层：交叉验证 - 关键信息用第二个模型或权威知识库交叉检查 - 如果两个来源矛盾，不输出结果，转人工 - 宁可不回答也不回答错

第四层：用户反馈闭环 - 用户可以标记"这个回答有问题"，立刻触发人工审核 - 这条 case 进入 bad case 库用于模型改进 - 同时追踪：有没有用户按照 AI 建议行动后出了问题？这是最严重的信号

跟我做 intern-clawd 的思路一致——宁可坦诚说不知道，不可自信地说错。 这个原则在医疗/金融这种高风险场景要贯彻到极致。

Q9: 给支付宝加 AI 板块怎么加？怎么评估

思考框架：产品定位→具体功能→评估体系。

产品定位： 不做一个独立的"AI 板块"跟理财保险并列，而是做一个贯穿所有场景的统一 AI 对话入口。用户点进去之后，根据他说的话路由到不同场景——说"上个月花了多少"走账单，说"帮我推荐理财"走理财，说"牙疼赔不赔"走保险。

这样做的好处是：用户不需要知道自己的需求属于哪个板块，说一句话就行。随着用户用得越多，这个 AI 入口积累的 Context 越深，体验越好，慢慢就会变成用户打开支付宝的默认入口。

具体功能（按优先级）： 1. 智能搜索/导航 — "怎么退款""花呗怎么还"，解决用户在 app 里找不到功能的问题 2. 消费洞察 — "我上个月花了多少外卖"，AI 直接说人话，不用用户自己看图表 3. 理财助手 — "我有 5 万闲钱怎么放"，给个性化建议 4. 保险顾问 — "我需要什么保险"，场景化推荐

评估体系（三层指标）：

层级 指标 说明
功能维度 AI 入口点击率、对话完成率、意图识别准确率 AI 本身好不好用
业务维度 通过 AI 产生的理财购买 GMV、保险转化率 AI 有没有带来业务增量
体验维度 用户满意度评分、次月留存变化 用户愿不愿意回来用

AB 测试设计： 有 AI 入口 vs 没有的用户对比。注意不能只看 AI 使用率——使用率高但理财购买没涨，说明用户在"玩"AI 而不是在"用"AI。要看穿透到业务的指标。

Q10: Prompt 调优技巧

思考框架：方法论分层，从最重要到最不重要。

我总结了五条实战中最有用的技巧，按重要性排：

1. 角色定义要精准（最重要） - "你是一个 AI 助手"效果很差 - "你是一个服务于蚂蚁保用户的保险顾问，用户大多是 25-35 岁、第一次买保险、对条款不熟悉"效果好得多 - 角色越具体，LLM 越知道该用什么视角、什么语气、什么深度 - 我的情报系统一开始角色定义太泛，改成"从 8 个核心视角提取 signal 的 AI 战略分析师"后产出质量明显提升

2. 给 few-shot 示例比给规则有效 - 写一大段"不要用主观语气、不要超过 50 字、不要用专业术语"，LLM 经常违反 - 直接给 2-3 个好的输出样例，LLM 模仿示例比理解规则准确得多 - 这是 few-shot learning 的核心价值——show don't tell

3. 复杂任务拆步骤 - 一个 prompt 同时做"筛选 200 条 + 提取 10 条 + 生成 insight"效果很差 - 拆成两步：第一步只做 signal 提取，第二步拿 signal 生成 insight - 每步目标单一，质量可控，出错了也知道是哪一步的问题

4. 用 bad case 驱动迭代，不凭感觉改 - 每轮攒 10-20 个 bad case，分析共性问题 - 比如我发现连续几天都漏了中国大厂动态 → 分析后发现是视角定义里没强调"含中国大厂" → 加上后马上好了 - Prompt 调优不是灵感驱动，是数据驱动

5. 温度和参数是最后调的（最不重要） - 很多人一上来调温度、调 top_p，其实 prompt 文本的影响力远大于参数 - Prompt 没写好，温度调到 0 也没用 - 参数只有在 prompt 已经足够好、需要微调输出多样性的时候才有意义

Q11: 常用的 AI 产品分析（Claude Code）

思考框架：做得好的 + 需改进的 + 对蚂蚁的启示。

我讲 Claude Code，因为我每天深度使用，还在它基础上搭了一套 Agent 系统。

做得好的： 1. 模型能力天花板高 — Opus 4.6 的推理和代码能力目前是顶级的，复杂任务一次成功率很高 2. 扩展点设计精妙 — hooks、CLAUDE.md、MCP 这些扩展机制让用户可以在不改源码的情况下深度定制行为。我就是靠这些扩展点把它改造成了秘书 3. 工具调用可靠 — 文件读写、命令执行这些基础工具的稳定性很好，几乎不出错

需要改进的： 1. 跨 session 失忆 — 每次新对话从零开始，不记得之前聊过什么。这是我做 intern-clawd 的出发点——给它装一套外挂 Memory 2. 没有主动能力 — 只能被动响应，不能主动提醒、主动执行定时任务。我通过 cron + 飞书 bot 补了这块 3. 多模态交互弱 — 不支持语音输入输出，移动端体验差

对蚂蚁的启示： 一个好的平台不需要什么都做到，只要扩展点设计得好，用户/开发者会自己补。对支付宝做 AI 也适用——与其做一个大而全的 AI 功能，不如把 AI 能力开放成 API 和 Agent 可调用的接口，让生态来补场景。

Q14: 蚂蚁 AI 核心竞争力

场景纵深 × 数据闭环 × 合规壁垒。

  1. 场景纵深：支付宝覆盖支付/信贷/保险/理财全场景，每个场景对 AI 的准确性、安全性要求极高。这个门槛把大部分竞争者挡在外面
  2. 数据闭环：用户在蚂蚁的行为数据越多 → AI 推荐越准 → 用户越依赖 → 产生更多数据。这是正向飞轮
  3. 合规壁垒：金融 AI 的合规要求比电商 AI 高一个数量级。蚂蚁在隐私计算、合规框架上的积累本身就是护城河

一句话：高门槛场景 + 最全数据 + 合规能力 = 别人想做也做不了。

Q15: Agent-to-Agent 的未来（⭐主动抛出的亮点观点）

我体验完蚂蚁保和天猫超市的 AI 之后有一个感受——现在的 AI 是散落在每个 app 里的，每个 app 都有自己的 AI 助手，但互相不通。

我认为终局是每个人有一个自己的 Agent，它有你最全的 Context，代表你去跟各个平台的 Agent 交互。你要买保险，不是你打开支付宝跟蚂蚁的 AI 聊，而是你的 Agent 拿着你的健康数据、家庭结构、财务状况，去跟蚂蚁的 Agent 谈。

这对蚂蚁既是威胁也是机会——威胁是用户不直接打开支付宝了；机会是谁的 Agent API 做得最好、数据最开放，谁就是其他 Agent 优先调用的服务商。

未来互联网公司的竞争不再是抢用户的眼球，而是抢用户 Agent 的优先级。

Q16: intern-clawd 技术实现——怎么用 CLAUDE.md + Hooks 搭的

思考框架：两个扩展点各解决什么问题。

Claude Code 有两个关键扩展点：

CLAUDE.md = Agent 的灵魂 - 项目级指令文件，Claude Code 每次启动自动读取 - 我在里面写了秘书的人格定义、行为规则、红线约束 - 比如"你是 boss 的私人秘书，偏好建议性表达"、"写 wiki 前必须确认" - 不管开多少 session，角色和准则始终一致

Hooks = 自动化的行为触发器 - SessionStart hook：每次新 session 启动，自动把 wiki index + 最近 log 注入。解决跨 session 失忆——Claude 一醒来就先"读笔记" - PostWrite hook：每次写完文件自动跑 wiki-lint，检查格式和交叉引用。自动化质量兜底

关键设计原则：没改 Claude Code 一行源码。 完全用原生扩展点在外面叠了一层记忆和仪式。就像给手机装壳——功能增强但手机本身没变。这也是跟 OpenClaw 的本质区别——OpenClaw 自己造新手机，我给已有手机装壳。

思考框架：按金融场景分类，每个对标蚂蚁的哪块业务。

全球范围内跟蚂蚁业务重合的 AI-native 金融产品：

1. Stripe Radar（支付风控） - 全球支付领域最成熟的 AI 反欺诈，ML 实时判断每笔交易 - 对标蚂蚁风控。区别：Stripe 主要 B 端，蚂蚁同时覆盖 C 端

2. Klarna AI 客服（信贷客服） - 瑞典先买后付公司，AI 客服替代 700 人工座席，处理 2/3 咨询 - 对标蚂蚁花呗/借呗客服。全球最早大规模落地 AI 客服的金融公司之一

3. Wealthfront / Betterment（智能投顾） - 用户输入目标和风险偏好，AI 自动配置组合并持续调仓 - 对标蚂蚁理财助手。区别：美国市场，中国监管和用户习惯不同

蚂蚁国际 AI 产品经理 — 补充题（国际业务深水区）

补充日期： 2026-04-17 场景： 蚂蚁国际 一面（15:45） 用途： 原备忘录（蚂蚁面试备忘录.md）国内业务覆盖充分，本文件补国际角度 10 题

一、业务理解（3 题）

Q18: 介绍一下 Alipay+ 的商业模式，它的核心竞争壁垒是什么？

思考框架：三方 network — 游客/商家/本地钱包，蚂蚁做连接器

Alipay+ 本质是一个跨境支付网络 API 层，连接全球 100+ 电子钱包（GCash / TrueMoney / Kakao Pay / Touch 'n Go）和全球商家 POS 系统。

商业模式： - 商家侧收手续费（比卡组织 2-3% 低，约 1-2%） - 钱包侧收 network fee - 外汇和清算赚点差

三层壁垒： 1. 两侧网络效应 — 接入的钱包越多，商家越愿意接；商家越多，钱包越愿意接。冷启动早，后来者很难追 2. 跨境清算基础设施 — 蚂蚁自己的 AntChain 做实时清算，T+0 结算比传统 SWIFT 快 90% 3. 中国游客流量 — 中国出境游每年 1.5 亿人次，Alipay+ 是海外商家吃中国游客流量的默认通道

AI 价值点：实时汇率预判、反欺诈（不同市场欺诈模式差异大）、商家智能推荐（哪款产品卖中国游客更好）

避坑：不要把 Alipay+ 说成"Alipay 的国际版"，它是全新网络，本质上是 Mastercard/Visa 的竞争对手

Q19: MyBank 的 310 模型（3分钟申请/1秒到账/0人工）在东南亚能复制吗？主要阻碍是什么？

思考框架：拆解 310 依赖什么 → 东南亚缺什么

310 在国内 work 依赖四块数据基础设施： 1. 阿里电商交易流水（商家视角） 2. 支付宝消费+收入数据（个人视角） 3. 全国征信系统（行为+履约） 4. 身份实名验证（身份证+人脸）

东南亚能复制的： - 新加坡 / 香港：征信系统完善，SME 贷款有成熟风控，可以复制 - 印尼：OJK 有基础征信但 coverage 低，需要蚂蚁拉电商+钱包数据补位

东南亚难复制的： - 菲律宾 / 越南：大量 SME 是现金经营，没有数字流水可做风控原料 - 各国身份验证系统不互通，KYC 成本高 - 当地中小银行合作博弈：他们不愿意把贷款风控外包给外国公司

AI 能解决的：用 LLM 处理非结构化数据补位（商户社交媒体、评价、WhatsApp 商家群聊），在征信薄弱市场做替代数据风控。这恰好是我 AFI 的技术栈方向

避坑：不要说"东南亚落后所以 work"，要说"数据基础设施不同，需要不同数据源"

Q20: WorldFirst 跟 Alipay+ 是什么关系？蚂蚁国际的整体产品版图怎么看？

思考框架：C 端 vs B 端 + 收 vs 付 四象限

蚂蚁国际其实是"支付四象限"的全覆盖：

C 端（个人） B 端（企业）
付出去 Alipay+ / AlipayHK —
收进来 Alipay+ 商家端 WorldFirst（出海卖家收款）
  • Alipay+ = 游客/个人跨境支付
  • AlipayHK = 香港本地 C 端
  • WorldFirst = 出海 B 端卖家（亚马逊 / eBay / Shopee 卖家从海外买家收钱）
  • MyBank 能力输出 = 给东南亚合作银行做 SME 贷款
  • AntChain = 底层清算 / 资产上链

整体叙事：蚂蚁国际不是"把 Alipay 卖到国外"，是把支付 + 信贷 + 清算 + 风控的能力拆成 API，卖给当地金融机构和跨境商家。从产品形态上类似 Stripe（B 端）+ Wise（跨境汇款）+ Visa（清算）的组合体

二、市场差异（2 题）

Q21: 东南亚用户 vs 中国用户，在金融 AI 产品上的关键差异是什么？

思考框架：设备 / 信任 / 支付习惯 / 文化 / 监管 五维度

1. 设备差异 - 印尼 95% + 手机端，大量低端 Android 机（2GB RAM，4G 网络） - → AI 功能必须轻量化，不能依赖大模型本地推理，要云端轻 API

2. 信任模式 - 东南亚用户对金融平台信任度低（历史上金融诈骗多） - → 透明度权重更高：AI 推荐必须给出处、给对比、给"为什么是你" - 中国用户习惯"小红书/抖音推啥我买啥"，AI 黑盒可接受

3. 支付习惯 - 印尼 / 菲律宾：大量现金交易，钱包是补充不是替代 - 中国：现金近乎消失 - → AI 导购场景不同，东南亚要覆盖"线下 + 少量线上"，中国纯线上

4. 语言 / 文化 - 东南亚多语言碎片化（印尼语 / 泰语 / 马来语 / 越南语），每种语言 LLM 语料少 - → 不能直接翻译中文 prompt，要按市场重写角色和 few-shot

5. 监管 - 每个国家 KYC/AML 标准不同，印尼有 PP 71 数据主权法 - → 合规规则引擎必须 per-market 硬编码，不能全局一套

我作为新加坡人，这些差异我日常能感知到，不是从 report 里读来的

Q22: 如果你看到 GCash 也在做 AI 理财顾问，你怎么定位 Alipay+ 跟 GCash 的关系？是合作还是竞争？

思考框架：合作 vs 竞争不是二选一，是层级问题

我认为是"同一市场的合作伙伴，不同层级的潜在竞争"：

现在：合作 - GCash 是 Alipay+ 的钱包合作伙伴，Alipay+ 帮 GCash 覆盖更多海外商家 - GCash 的 AI 理财顾问服务菲律宾本地用户，不威胁 Alipay+

中长期：可能竞争 - 如果 GCash 开始做跨境 API（菲律宾用户在泰国消费），就进入 Alipay+ 的领地 - 如果 Alipay+ 开始做本地钱包（在菲律宾直接做 C 端），就跟 GCash 对撞

战略判断：蚂蚁的聪明选择是继续做"连接层"不做"钱包层"，让本地钱包们分散、互相牵制，蚂蚁做唯一的跨境连接器。就像 Stripe 不做银行卡，只做支付基础设施

对 PM 工作的 implication：跟本地钱包合作方的产品设计，要避免"让他们感觉在被蚕食"。API 开放度、数据共享规则、收入分成，都要经过 politics sensitivity 过滤

三、本地化与 AI（2 题）

Q23: 多语言多市场的 AI Agent，跟只做中文的 AI 产品，最大的设计挑战是什么？

思考框架：3 层差异 — prompt / 数据 / 评估

挑战一：Prompt 本地化 ≠ 翻译 - 翻译会丢语气和文化 cue - "帮我推荐一款保险" 在泰国可能要问"你全家几个小孩"（家庭金融决策权在女性），在印尼要问"你的家庭是穆斯林吗"（Sharia-compliant 金融产品） - → 需要每个市场独立设计 system prompt 和 few-shot，不能全球一套翻译

挑战二：训练/评估数据稀缺 - 中文金融 QA 数据几十亿 token，印尼语可能只有百万 token - LLM 在小语种金融场景上幻觉率显著高 - → 要用多语言+领域知识双重 fine-tune，或者保守策略"不确定就切换到英语回答"

挑战三：评估体系不能共用 - 中国的 "prompt → output → 人工标" 流程在东南亚面临人工成本和语言障碍 - → 要建分布式评估网络，每个市场找本地 PM/合规顾问做 bad case review - 我的 AFI 项目里做过类似设计：每条 signal 必须来源可溯，评估人不用读原文也能判准确性

避坑：不要说"翻译一下就行了"，会被判为没做过国际化

Q24: 印尼用户用 2GB 内存的低端手机，你要给他们做一个 AI 理财助手，技术和产品上怎么权衡？

思考框架：技术架构 + 体验降级 + 场景取舍

技术层面 - 不能依赖本地大模型，所有推理走云端 API - 页面轻量化：不用 React/Vue 重前端，原生 HTML + 少量 JS - 图片用 WebP + CDN 边缘节点（CloudFront 东南亚 POP） - AI 响应流式输出，让用户 200ms 内看到第一个字

产品层面 - 功能精简：只做 3 个核心场景（看余额、推荐储蓄产品、记账助手），不堆复杂功能 - 对话长度限制：每次对话最多 5 轮（节省 token 成本 + 用户也不习惯长对话） - 离线降级：没网时显示缓存的推荐，"等联网后更新"

场景取舍 - 放弃：复杂投资组合、股票分析（印尼用户不是主力） - 保留：储蓄产品推荐、信贷申请、日常消费分析 - 强化：账单提醒 + 家庭共享（东南亚用户家庭资金决策多）

底层原则：东南亚市场是 good enough 胜过 feature-rich。中国团队容易把功能堆太多，需要 PM 强制做减法

四、合规与竞品（3 题）

Q25: 数据主权（data sovereignty）在跨境 AI 产品里是个硬问题。印尼 PP 71、越南 Decree 53 都要求本地化存储。你怎么处理？

思考框架：法规硬约束 + 架构设计

核心原则：用户数据不能跨境，但可以用蒸馏后的模型参数跨境

架构设计： 1. 数据本地化：用户原始数据（交易流水、对话记录）存印尼本地 AWS / 阿里云节点，不出境 2. 推理本地化：AI 推理跑在本地节点，prompt + response 都不出印尼 3. 模型训练分层： - 全球通用模型（金融基础知识、语言理解）→ 中国/全球训练 - 本地 fine-tune 用联邦学习：只把梯度上传，原始数据不动 4. 审计日志本地留存：监管随时可查

产品 trade-off：本地化推理延迟比中心化高，功能发布节奏慢。但这是法规硬成本，必须接受

我的相关经验：intern-clawd 设计时就遵循"数据留在用户本地"原则 — wiki 存在用户自己的 ~/.clawd/ 里，不上传云端。这跟跨境合规的 mindset 一致

Q26: Wise / Revolut / Stripe 对蚂蚁国际构成什么威胁？你怎么看竞争格局？

思考框架：按业务拆 + 差异化优势

不是铁板一块的威胁，是分场景的竞争：

Wise（跨境汇款，C 端） - 优势：汇率透明、费用低、面向欧美用户强 - 威胁 Alipay+ 的 C 端跨境支付 - 蚂蚁反制：亚太用户规模 + 商家网络 Wise 追不上

Revolut（新银行 + 投资 + 跨境） - 优势：欧洲用户基数大，金融产品深度 - 威胁：直接和 Alipay+/MyBank 对标亚太中产 - 蚂蚁反制：跟中国游客的强绑定是 Revolut 的盲区

Stripe（B 端支付基础设施） - 优势：全球开发者默认支付 API - 威胁 WorldFirst 的 B 端跨境收款 - 蚂蚁反制：中国出海卖家是 Stripe 覆盖弱的部分，蚂蚁本土优势明显

AI 层面的差异化： - Stripe Radar 是业界反欺诈最强，但只做支付风控 - Wise 没什么 AI 产品 - Revolut 有 AI 理财助手但只覆盖欧洲 - 蚂蚁的机会：把 AI 能力跨产品打通（支付→理财→信贷都能用同一个 Context），这是垂直整合的优势

一句话：蚂蚁国际的护城河是"亚太 + 华人 + 多产品整合"，不要在欧美市场硬碰 Wise/Revolut

Q27: 如果让你给 WorldFirst 设计一个 AI 功能服务中国出海卖家，你会做什么？

思考框架：用户画像 → 痛点 → 最小 MVP

用户画像： - 中国中小电商卖家（Shopee / Amazon / TikTok Shop 卖家） - 英语不好，海外平台规则看不懂 - 多平台多币种回款，账目混乱 - 汇率波动焦虑，不知道何时换汇

三个痛点对应三个 AI 功能：

1. 平台规则 AI 顾问（解决"看不懂英文"） - 输入：Shopee/Amazon 的新规邮件（英文） - 输出：中文摘要 + 对你店铺的具体影响 + 需要做什么 - 价值：省下卖家每天 30 分钟翻译规则的时间

2. 多账户对账助手（解决"账目混乱"） - 自动汇总 Shopee/Amazon/eBay 各币种回款 - AI 标注异常（这笔钱和订单对不上） - 导出给会计的标准化报表

3. 智能换汇建议（解决"汇率焦虑"） - 基于历史波动 + 即期新闻 + 你的现金流需求 - 给"建议现在换 / 再等等"的简单提示 - 不是预测汇率（合规风险），是给决策参考

MVP 选：先做 #1 平台规则 AI 顾问，因为最刚需、合规最安全（只翻译解释，不做投资建议）、冷启动最快（英文→中文翻译是 LLM 强项）

五、战略与组织（1 题）

Q28: 如果你是蚂蚁国际 AI PM，要同时管 3 个国家（新加坡、印尼、泰国）团队迭代同一个产品，组织上最大的挑战是什么？

思考框架：不是技术问题，是人的问题

三个层次的挑战，难度递增：

第一层：时区（相对容易） - 三国时差 1-2 小时，还 OK - 解决：每周固定时间 all-hands，其他用异步文档同步

第二层：市场差异导致的需求冲突（中等） - 印尼团队说"要支持穆斯林 Sharia 金融产品"，泰国团队说"没必要" - 解决：产品平台抽象 — 配置化而非硬编码，每个市场自己开关功能

第三层：文化和决策权博弈（最难） - 中国总部 PM vs 当地 country head 的博弈：总部想统一、本地想差异化 - 解决思路： - 清晰定义全局 vs 本地边界：技术架构、合规框架、品牌调性 = 全局；UI 语言、功能取舍、营销活动 = 本地 - 把本地 PM 纳入决策链，而不是"总部定了让本地执行" - 数据说话：每个 feature 的 A/B test 分国家跑，用数据而非权威来判

个人认知：我作为新加坡人跟过这种跨境项目（CUHK 的跨区域经济合作研究），我知道"平衡"不是妥协到中间，是找到"哪些事必须统一，哪些事必须允许分叉"的分界线

六、可能的 Curveball（开放题）

Q29: 你觉得蚂蚁国际这个"连接本地钱包"的战略，有什么短板？如果你是战略层，你会建议怎么补？

思考框架：坦诚指出问题 + 给出方向（不卑不亢）

短板： 1. 不拥有 C 端用户，依赖本地钱包的用户关系。本地钱包一旦自己做跨境（GCash 已经开始了），蚂蚁的 value 被稀释 2. 品牌认知弱：用户用的是 GCash / TrueMoney，不知道背后是 Alipay+。没法直接做品牌杠杆 3. 数据碎片：每个本地钱包只给蚂蚁有限数据，蚂蚁拿不到全量用户行为

可能的补法： 1. 往下做本地钱包（收购 or 自建）—— 风险大、资本重 2. 往上做商家生态（给全球商家做"一站式多钱包 SDK"）—— 深化 B 端锁定 3. 往旁做 SaaS 赋能（给本地钱包卖 AI 能力 / 风控 / 合规工具）—— 用 AI 当新的粘性点

我倾向 #2 + #3 组合：不做 C 端直接碰瓷本地伙伴，但通过 B 端和 SaaS 让自己变成基础设施不可替代

这题的亮点：坦承战略有弱点，但给的补法不破坏现有合作方关系

二、高概率英文问题（5 题 kernel）

Q-E1: Walk me through your most recent project in detail.

Framework: Problem → Solution → Impact → Learning (STAR-like)

The one I'm proudest of is the AI Frontier Insight system. The problem was that our tech leadership at Xiaohongshu was drowning in AI news — TechCrunch, arXiv, Twitter, GitHub trending, Hugging Face — and nobody had time to synthesize it into something actionable.

So I built a pipeline with three stages: collection (parallel RSS plus Twitter List plus GitHub plus arXiv plus HF), signal extraction using DeepSeek to distill 200 raw items into 10 signals, and insight generation to turn those signals into strategic implications for our business.

The interesting part wasn't the tech — it was the prompt engineering. I did four iteration cycles, each driven by bad cases. For example, early versions kept missing China's domestic AI landscape. I dug into the bad cases and realized my role definition was too Western-skewed. Once I rewrote the system prompt to explicitly include "China's major AI labs", coverage fixed itself in one day.

Impact: it's been running daily for about three months now, and several of our VPs have quoted insights from it in their weekly meetings. That's probably the strongest validation.

Vocabulary to nail: - "distill" (蒸馏) — better than "extract" for prompt context - "bad case driven iteration" — 中文"bad case 驱动迭代"的英文对应 - "role definition" — prompt engineering term - "strategic implications" — 战略 implications

Q-E2: Why Ant International specifically? Why not somewhere else?

Framework: Personal fit + Unique value

Three reasons, and they stack:

First, the problem space excites me. Cross-border fintech in Asia is one of the few places where you're genuinely solving hard problems — regulatory fragmentation, infrastructure gaps, cultural localization — not just iterating on existing UX patterns. That's meaningful work.

Second, my background is uniquely suited. I'm Singaporean — I grew up with DBS, GrabPay, and Alipay HK in the same wallet. I studied in Hong Kong and Chicago, so I'm comfortable operating across Chinese and English professional contexts. Most candidates can offer one of these; few can offer both.

Third, the team profile matches how I work. From what I've read about Ant International's AI team, the focus is on building production systems that serve real users at scale, not shipping demos. My projects have always been production-oriented — that's where I learn most.

Candidly, I've looked at Grab Financial, Revolut, and Wise — but none have the combination of Asia-first focus, multi-product depth, and AI maturity that Ant has.

注意: 提 Grab / Revolut / Wise 展示你做过竞品功课，是加分点

Q-E3: What's the hardest technical challenge in building AI for multi-market financial products?

Framework: 3 layers — data / model / evaluation

I'd say it's evaluation, counterintuitively. Everyone talks about prompt engineering and model fine-tuning, but in a multi-market financial context, the real bottleneck is: how do you know your AI is actually behaving correctly in Indonesian, Thai, and Vietnamese?

Data layer: Non-English financial QA datasets are orders of magnitude smaller than Chinese or English. Your model just knows less.

Model layer: Even if you have a great model, hallucinations in low-resource languages are systematically higher — and in finance, a hallucinated recommendation is a real liability.

Evaluation layer: You can't just translate your English eval set. A question like "is this insurance Sharia-compliant" doesn't exist in your English dataset at all. You need localized evaluation networks — real PMs and compliance experts in each market who can flag bad cases in their language.

I actually faced a mini version of this in my intelligence system project. My early evaluation was "do I think this signal is good" — totally insufficient. I ended up building a rubric with seven explicit criteria, so anyone could grade consistently. That same rigor has to scale across markets.

Q-E4: Tell me about a time you had to push back on a stakeholder.

Framework: STAR (Situation, Task, Action, Result)

At Xiaohongshu, my manager asked me to generate a weekly AI trends report for our executive team. The initial ask was: "write a summary of what's happening in AI this week."

I pushed back, not on the output, but on the framing. I said: "If we just summarize news, we're creating noise. What our VPs actually need is to know which trends affect our business and what we should do about it."

So instead of a generic report, I built a two-layer output: a daily signal feed that's informational, and a weekly strategic insight that's actionable — each signal explicitly mapped to which of our business lines it affects.

The result: our CTO started quoting the weekly insights in all-hands meetings. That wouldn't have happened with a generic summary.

The learning for me: pushing back on framing is often more valuable than pushing back on execution. It's harder because it requires you to propose a better alternative, not just flag a problem.

Q-E5: What's a product decision you would make differently if you could go back?

Framework: Genuine regret + what you learned

Early in the intelligence system, I optimized for coverage — I wanted to pull from every possible source: 10+ RSS feeds, three Twitter lists, arXiv, GitHub, Hugging Face, benchmarks. More sources equals more signal, right?

Wrong. What I got was noise, not signal. The quality of my daily brief got worse as I added sources, because my LLM had to filter more low-signal-to-noise content.

What I should have done: start with three high-quality sources (Twitter from top researchers, one curated newsletter, arXiv abstracts), nail the pipeline, then add sources only when there was a clear coverage gap.

The lesson: breadth is a vanity metric. In product, I'd rather do 3 things excellently than 10 things adequately. That's something I'd bring to whatever I build here.

Why this works: Shows self-awareness, shows you can kill your darlings, and ties back to good product instinct

六、常见陷阱 + 应对

Trap 1: Answering too fast in English

  • Why: 想证明自己英文好，反而显得紧张
  • Fix: 每句话后主动停顿 0.5 秒，让面试官跟上

Trap 2: Translating Chinese sentence structure literally

  • "我认为这个产品的价值主要体现在..." → "I think the value of this product mainly is reflected in..." ❌
  • Better: "The core value is..."
  • Fix: 用英文句子起头重写，不要翻译

Trap 3: Over-using "very" and "really"

  • "It's very interesting" ❌ weak
  • "It's a fascinating problem" or "That's exactly the kind of question I enjoy" ✅
  • 美式面试偏好具体形容词，少用程度副词

Trap 4: "Actually" as filler

  • "Actually, I think... Actually we did... Actually..."
  • 每句开头 actually 会显得 defensive
  • Fix: 只在真的要反驳时用 actually

Trap 5: Drop into Chinese mid-sentence

  • "The 核心价值 is that it solves..." ❌
  • 即使英文单词不记得，也用描述 + 英文解释而不是直接切换
  • Good: "The core value proposition — what we'd call 核心价值 in Chinese — is..."

Trap 6: Can't describe AFI / intern-clawd in English

  • 练一次 30 秒 elevator pitch 用英文描述每个核心项目
  • 提前准备好核心数据的英文表达："200 items daily", "four iteration cycles", "50-plus GitHub stars"

七、如果面试官英文一般（中国面试官居多）

有可能面试官英文水平不高，但又要 test 你。策略： 1. 你的英文放慢 20%，措辞偏简单，不用长从句 2. 复杂概念用中文辅助解释："In Chinese we'd say 护城河, which in English is 'moat' or 'defensibility'" 3. 别让他感到自己英文差（他可能因此给你低分）

八、最后 5 分钟念一遍的 3 句话

进面试前，英文 self-intro 念 3 遍。重点这三句定调：

  1. Opening: "Hi Kevin, I'm Amber, Singaporean..." (温度 + 身份卡)
  2. Hook: "I built [AFI] end-to-end, from requirement to production..." (主动性 + 结果导向)
  3. Closing: "...especially for Ant International, where firsthand Southeast Asian perspective is rare." (点到为止，把加分点钉上)

这三句念熟了，英文环节开场稳住，后面即使磕巴也不慌。

🏛️ Layer 1 · MaaS 中台（对应蚂蚁百灵）

定位：Model as a Service，把大模型能力打包成 API，给内部 BU + 外部开发者调用。
对标：阿里通义百炼 / 字节火山方舟 / OpenAI API / Anthropic API
蚂蚁特殊性：金融场景 + 东南亚市场 + 跨境合规
L1-Q1. 蚂蚁百灵和通义、火山方舟、OpenAI 怎么差异化？
公式：不在通用能力拼算力，在垂域 + 合规 + 本地化找差异化
  • 垂域：金融垂域（风控 / 反洗钱 / 合规 / 理财术语）——通用大模型都弱，蚂蚁有独家场景数据
  • 合规：跨境金融的监管 / 数据主权 / 反诈等，蚂蚁国际做过十年，积累了合规能力
  • 本地化：东南亚语种 fine-tune，尤其印尼、泰、越——通义、OpenAI 都没深做
加分："通用大模型的竞争已经是算力竞赛，垂域 + 合规 + 本地化三维组合不是谁钱多谁赢，是数据、场景、关系网络的护城河。蚂蚁三样都有。"
L1-Q2. 如果让你负责 MaaS 的产品定义，第一步做什么？
公式：先定客户，再定场景，再定能力。不要一上来做"通用大模型"
  • 客户选择：对内（蚂蚁系 BU）优先，对外（第三方金融机构）次之——先让内部产生价值闭环，再外部化
  • 场景选择：选 3 个高频刚需场景（如：风控报告生成 / 客服问答 / 合规文本审查），快速验证
  • 能力封装：不是给 API，是给场景化能力卡（"风控助手"、"合规审查助手"），降低 BU 接入门槛
加分：引阿里百炼经验——"百炼早期做通用 API，内部接入慢；后来做场景化能力卡，接入速度 10x"。
L1-Q3. 东南亚语种投入：自研 vs 基于开源微调？
公式：基于开源微调是首选，不要自研
  • 开源（Llama / Qwen）已经覆盖印尼、泰、越基础能力，差的是金融垂域
  • 自研 base model 烧钱 + 周期长，ROI 远低于 fine-tune
  • 关键投入在数据工程：收集高质量东南亚金融语料 + 指令数据集 + 人类反馈
  • 自研值得做的只有一件事：东南亚语言的专用 tokenizer（通用 tokenizer 对泰语、越南语切词差，影响效果）
L1-Q4. MaaS 的关键指标是什么？
公式：分三层——规模 / 质量 / 商业
层指标说明
规模API 调用量 / 接入的 BU 数 / MAU广度
质量场景任务准确率 / latency / 幻觉率深度
商业平均成本 per call / 对内节省的人力成本价值
"北极星指标我会选跨 BU 调用量 × 平均任务完成质量的组合——既看广度又不让人刷量。"
L1-Q5. 金融垂域微调值不值得做？
公式：值得，但要选场景
  • 值得做的场景：风控规则解释 / 反洗钱流水分析 / 合规文本审查 / KYC——这些场景通用大模型准确率只有 60-70%，微调能到 90%+
  • 不值得做的场景：客服闲聊 / 通用翻译——通用大模型已经够好
  • 成本：一次微调投入约 X 万，但服务几十万企业用户，ROI 高
L1-Q6. 模型幻觉 / 安全 / 合规怎么处理？（金融场景必问）
公式：不能信任模型，要做外部校验
  • 幻觉：金融场景下零容忍——架构上用 RAG（检索外部事实）+ 拒答机制（模型没把握就不说）
  • 安全：prompt 注入防御 / PII 脱敏 / 输出过滤 三层
  • 合规：审计日志必须全量留存；关键决策链路可追溯；输出打水印
"我在 AFI 项目里踩过幻觉的坑——LLM 会编造没见过的模型发布。后来加了 source citation 要求，模型必须引用原文才给输出。这是金融场景的必备设计。"

定位：内部 Copilot，给员工（研发 / 运营 / 商家服务 / 客服）提效。
对标：Cursor / GitHub Copilot / 飞书智能伙伴 / Shopee 内部员工 AI 导购
蚂蚁特殊性：跨国研发团队 + 金融合规 + 多语言员工
L2-Q1. 蚂蚁内部员工 AI Copilot 应该先从哪类员工开始？
公式：高频重复 × 可量化产出的岗位优先
  • 研发（最优先）：代码补全 / code review / 单元测试生成。产出可量化（PR 数 / bug 率）
  • 客服（次优先）：工单回复 / FAQ 生成。产出可量化（响应时长 / 解决率）
  • 运营（第三）：报告生成 / 数据分析。产出半可量化
  • BD / 法务（暂缓）：高价值但难量化，先等前三项跑通
L2-Q2. 怎么衡量 Copilot 的效果？
公式：采纳率 + 节省时间 + 质量 + 持续使用率四维
  • 采纳率：生成的建议用户接受的比例（研发场景看 AI 生成代码被 keep 的比例）
  • 节省时间：同样任务 AI vs 人工的时长对比（需要 baseline 测量）
  • 质量：AI 产出的 bug 率 / 客诉率 vs 人工基线
  • 持续使用率：最关键——采纳率高但 1 个月后没人用就是新鲜感
"新鲜感 vs 真价值的判断方法见阿里 Q30——第二周新用户数据还好才说明真有价值。"
L2-Q3. 跨语言代码注释 / code review 有什么特殊挑战？
公式：语言不是主要问题，context 不一致才是
  • 研发团队跨中印尼越，注释里中英印尼混用——通用 Copilot 处理不好
  • 更深的问题：跨团队 context 不对齐——一个团队的"common sense"在另一个团队不成立。AI 要有团队级 context 知识库
  • 解决：团队级 MEMORY（类似 intern-clawd 的 wiki），让 AI 知道这个项目的 convention / 业务术语 / 合规要求
"我做 intern-clawd 的核心思想就是这个——Claude Code 默认没有项目 context，加了 wiki 就懂了。蚂蚁国际跨团队 Copilot 可以复用这个思路。"
L2-Q4. 合规 code review 怎么做？
公式：规则引擎 + LLM 两层
  • 第一层（规则引擎）：硬性合规规则（反洗钱 API 必须调用 / 用户 PII 字段必须加密 / 特定操作必须记审计日志）——用静态分析 + 正则拦截
  • 第二层（LLM）：规则覆盖不到的"软合规"——比如代码意图不合规、绕过风控逻辑、数据使用范围超出授权。LLM 做意图理解
  • 两层都过才算通过
L2-Q5. 非技术员工（运营/客服）的 Copilot 产品形态？
公式：嵌入工作流，不要独立 app
  • ❌ 给客服一个 "AI 助手" app 让他切换使用——切换成本高，采纳率低
  • ✅ 在客服工单系统里直接嵌入"建议回复" button，点一下生成，改一下发送
  • 关键原则：Copilot 应该是工具的增强，不是新工具
"这个道理来自 Cursor 的产品哲学——不做新的 IDE，而是在 VSCode 基础上做增强。同样的思路搬到运营/客服场景。"
L2-Q6. 怎么说服高管继续投入研发提效？
公式：不用"效率提升 X%"这种虚指标，用"减少 HC 需求"
  • 虚指标：效率提升 20% → 高管不信（怎么量的？）
  • 实指标：同样业务体量下，研发团队本来要扩招 50 人，现在只需要扩招 30 人——省了 20 人的招聘成本 + 管理成本
  • 把 AI 投入换算成"等价人力"，高管立刻听懂

🧱 Layer 3 · 类 Dify（AI 应用搭建平台 for 内部）

定位：降低各 BU 搭 AI 应用的门槛，把公用能力平台化。
对标：Dify / 扣子 Coze / Langflow
蚂蚁特殊性：多 BU + 金融合规模板 + 多语言多国家
L3-Q1. 为什么要做平台，不让各 BU 自己搭？
公式：去重 + 治理 + 合规
  • 去重：每个 BU 都自己搭"风控检查"、"身份核验"，浪费 10 倍工作量
  • 治理：AI 应用不统一管，线上会出问题没人追责；平台统一审计日志
  • 合规：金融场景必须有一套合规模板，不能让每个 BU 自己理解合规（出错代价大）
L3-Q2. 平台应该提供哪些原子能力？
公式：蚂蚁金融场景的"公共抽屉"
  • 风控：用户风险分、交易风控、反洗钱检查
  • 身份：KYC、人脸识别、证件 OCR
  • 金融：汇率换算、利率计算、还款计划生成
  • 合规：PII 脱敏、合规文本审查、审计日志
  • 国际化：多语言翻译、时区 / 货币 / 法律规则切换
"这 5 类能力是蚂蚁国际每个 BU 都会用的，抽出来做平台能力接入成本从 2 周压到 2 天。"
L3-Q3. 怎么衡量平台价值？
公式：应用数 × 跨 BU 复用率 × 人均搭建时间
  • 应用数：平台上活跃应用的数量（绝对值）
  • 跨 BU 复用率：一个应用被几个 BU 复用（广度）
  • 人均搭建时间：从"想法"到"上线"的平均周期（效率）
  • 反指标：平台上线后，各 BU 还在自己写 AI 应用的比例——这说明平台没用
L3-Q4. 非技术人员能上手吗？门槛定在哪？
公式：要分层——不要所有人都能搭
  • Tier 1（高阶）：BU 里的技术人员/产品经理 → 可以组合复杂 workflow、调参数、接自己的数据源
  • Tier 2（中阶）：BU 运营 / 商家服务 → 只能基于模板修改，不能自创 workflow
  • Tier 3（初阶）：通用员工 → 只能用官方商店里的成品应用
  • 门槛低了乱了，门槛高了没人用——分层是解法
L3-Q5. 多语言 / 多合规模板怎么实现？
公式：把"国家"变成平台的一个变量
  • 应用搭建时选国家 → 自动加载对应的语言模板、合规规则、货币单位
  • 平台统一维护国家矩阵（印尼 / 泰 / 越 / 马来 / 菲 / 新 / 港）
  • 应用迁移到新国家 → 一键切换，不用重写
"这是蚂蚁国际做多国业务的结构性优势——把国际化从'每个应用的痛苦'变成'平台的能力'。"
L3-Q6. 外部化（像 Coze 那样对外）值不值得？
公式：先内部跑通，再考虑外部
  • 内部没跑通就外部化 = 把家丑外传
  • 外部化前提：内部至少 50+ BU 应用上线 + 3 个典型最佳实践可宣传
  • 外部化价值：扩大 ecosystem + 获取外部反馈 + 商业化（SaaS 模式）
  • 风险：对外开放意味着合规 / 安全要求更高，投入翻倍

📱 Layer 4 · toC 产品（Alipay+ · 跨境汇款 · 理财 · 客服）

定位：终端消费者直接使用的 AI 产品。
对标：支付宝 AI 搜索 / 微信 AI 助手 / Shopee AI 导购 / Klarna AI
蚂蚁特殊性：金融场景 + 东南亚低 AI 素养用户 + 多语言
L4-Q1. 蚂蚁国际最应该做的 toC AI 产品是什么？
公式：高频 × 刚需 × AI 独特价值三条全满足
  • AI 跨境汇款 Agent（首推）：用户一句话"给我妈妈转 500 RMB 到印尼"——Agent 自动处理汇率 / 手续费比较 / 合规审查 / 确认。高频（东南亚打工者刚需）× AI 独特（传统跨境汇款要填 15 个字段）
  • AI 客服（必做）：金融场景客诉多，AI 能接 50% 简单工单，省钱又提速
  • AI 理财顾问（中期）：需要用户信任，不能早做
  • AI 风险预警（差异化）：帮用户识别诈骗链接 / 异常交易，东南亚反诈刚需
L4-Q2. 东南亚用户 AI 素养低，交互怎么设计？（来自今早 Shopee 情报）
公式：多模态优先 + 意图澄清 chip + 容错兜底
  • 多模态：语音输入 >> 打字输入（打字门槛高）；拍照输入（付款码 / 证件 / 账单）
  • 意图澄清 chip：不要空白输入框，给用户常见需求的快捷入口（"转账"、"查余额"、"充话费"）
  • 容错兜底：AI 没把握时不要硬上，转人工客服；关键操作（转账）必须双重确认
"这三条不是我编的——今早 Shopee 一面面试官说东南亚 AI 搜索的三个特有约束（模型弱 / 用户素养低 / query 差），在蚂蚁国际金融场景只会更严重。"
L4-Q3. 金融场景的 AI 错了怎么办？
公式：分级权限 + 透明沟通 + 回滚机制
  • 分级权限：低风险（查询）AI 全自动；中风险（汇款）确认后执行；高风险（改密码）禁止 AI 操作
  • 透明沟通：AI 每次决策都告诉用户它在做什么 + 基于什么信息。不要黑盒
  • 回滚机制：汇款有 X 秒撤回窗口；订单有退订路径；错误有客服兜底
"核心原则：宁可多问一步，不可自作主张——金融场景的信任成本比体验流畅度更高。"
L4-Q4. 多模态入口（语音 / 拍立搜）的 ROI？
公式：东南亚 ROI 比国内高一个数量级
  • 国内用户打字门槛低（输入法成熟、用户习惯）→ 语音/拍照是锦上添花
  • 东南亚用户打字门槛高（多语言切换、拼写错误多）→ 语音/拍照是刚需替代
  • 优先做：拍照（账单 OCR、付款码扫描、身份证 OCR）——收益最直接
  • 中期：语音输入（需要东南亚语种 ASR 能力，投入周期长）
L4-Q5. 怎么和本地支付对手（GoPay / GrabPay / Touch'n Go）竞争？
公式：蚂蚁国际的"跨境"是别人学不会的
  • GoPay / GrabPay：本地支付强，但跨境场景（东南亚打工者给国内家人汇款）弱
  • 蚂蚁国际：有支付宝全球网络 + 合规能力——跨境是护城河
  • AI 切入点：AI Agent 把跨境场景做到极致流畅——本地对手短期内跟不上
  • 本地场景：不正面竞争，通过合作（GCash、Paytm、Alipay+）接入即可
L4-Q6. 用户画像 + Context 积累怎么做？
公式：显性 + 隐性 + 主动询问三层
  • 显性：注册时的国家、母语、职业等——用户主动填
  • 隐性：交易行为、支付场景、汇款目的——系统被动采集
  • 主动询问：AI 对话中获取"画像挖不到的信息"——比如"我每月要给家里汇一次钱"
  • 关键：AI 对话中采集的 Context 要回写到画像系统，形成闭环，让所有 BU 都能用
"Context 是护城河——用户在蚂蚁国际积累的 Context 越多，Agent 在这里表现越好，迁移到其他平台就要从零开始。这是我在 intern-clawd 里讲的'Context 红利'，在平台侧同样成立。"

🎯 跨层通用题（面试官可能用这个总结）

X-Q1. 四层你最想做哪一层？为什么？
公式：诚实选一层 + 给 3 条理由 + 承认其他层也重要
"我最想做 Layer 4 toC 产品，原因三点：
① 我做过 AFI 和 intern-clawd，都是把 AI 能力交付给终端用户的实践——我懂用户侧；
② toC 产品直接面对东南亚低 AI 素养用户，挑战最大也最有价值——我今早刚在 Shopee 面试听到这三个约束，有热情解决；
③ 其他三层（MaaS、提效、平台）是基础设施，我能理解它们的价值，但让我兴奋的是在基础设施之上为真实用户创造产品。

如果团队需要我去做 Layer 1 MaaS 或 Layer 3 平台，我也愿意——但我会主动带着 toC 场景的 sense 去看基础设施，反向提需求。"
X-Q2. 四层的资源优先级应该怎么排？
公式：基础设施先行，toC 跟上，提效在中
  • 短期（6 个月）：MaaS (L1) + 类 Dify (L3)——没有基础设施，上面什么都做不起来
  • 中期（1 年）：toC (L4)——有了基础设施后，快速产出可见价值
  • 长期（2 年）：类 OpenClaude (L2)——内部提效是"锦上添花"，资源紧时可以押后
"这是我的判断。我知道团队可能有不同看法——比如有人觉得 Layer 2 提效能快速见效应该先做。我想请教你们是怎么排的？"
X-Q3. 四层中哪一层最容易被竞对超越？
公式：MaaS 层最容易被超越，toC + Context 最难被超越
  • MaaS：模型层竞争激烈，阿里、字节、OpenAI 投入大，蚂蚁很难在通用能力上领先
  • toC + Context：最难被超越——用户在蚂蚁国际积累的交易数据、偏好、关系网络是独家资产，对手要从零积累
  • 结论：蚂蚁的长期护城河不在 Layer 1，而在 Layer 4 + Context
