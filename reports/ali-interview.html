三、Agent 知识备考（重点）

3.1 Agent 是什么（30 秒版）

Agent = LLM + 工具调用 + 记忆 + 规划能力。不是聊天机器人，是能自主完成多步骤任务的 AI 系统。

核心循环：感知 → 推理 → 行动 → 观察结果 → 继续（ReAct Loop）

3.2 Agent 核心组件

组件 作用 天猫超市场景举例
LLM（大脑） 理解意图、推理决策 理解用户”给我家猫买点吃的，要性价比高的”
Tool Use（手脚） 调用 API、查数据库、执行操作 调用商品搜索 API、比价接口、下单接口
Memory（记忆） 记住用户偏好、历史行为 记住用户养的是英短、之前买过某品牌
Planning（规划） 分解复杂任务为子步骤 “推荐猫粮”→ 确认猫种 → 筛选品牌 → 比价 → 推荐

3.3 Agent 架构流派（展示深度）

流派 代表 核心思路
Agent Loop 中心 Hermes Agent Agent 的执行循环是一切的起点，自动生成 skill
Gateway 中心 OpenClaw 消息路由和权限控制是核心，Agent 是组件之一
Harness 增强 intern-clawd 在现有 LLM 上叠加记忆+仪式+入口，不替换底层

你可以说：“我研究过主流 Agent 框架的架构差异——Hermes Agent 走 agent-loop-centric 路线，靠自动生成 skill 实现越用越聪明；OpenClaw 走 gateway-centric 路线，靠多平台覆盖建立护城河。我自己做的 intern-clawd 走的是第三条路：不做底层 agent，而是在 Claude Code 上叠加结构化记忆和仪式，让它从 coding 工具变成个人秘书。”

Q3: Agent 落地最大的挑战是什么？

“三个层面： 1. 可靠性——Agent 多步执行中一步错可能全盘崩，需要 human-in-the-loop 和 fallback 机制 2. 评估困难——不像搜索有明确的 NDCG，Agent 的好坏很难量化 3. 成本——每次对话消耗大量 token，需要在体验和成本之间找平衡（prompt caching、渐进式加载等）”

Q4: 你怎么理解产品经理这个角色？

“产品经理的核心职责是在用户价值和商业价值之间找到最优解。不是写 PRD 的人，而是定义’做什么、为什么做、怎么衡量做得好不好’的人。具体来说：发现真实需求（不是用户说什么就做什么）、定义优先级（资源永远不够，选最值得做的）、推动跨团队执行、用数据验证效果并迭代。”

六、面试前自检清单

七、Mock 面试题整理

Q1: 天猫超市最适合用 Agent 解决的问题是什么？

核心思路：天猫超市是高频、低决策成本、复购驱动（日用快消标品）。用户心智是”快速补货”不是”慢慢逛”。最适合管家型 Agent——记住购买周期、主动提醒、价格决策、一键下单。指标看月复购频次、Agent 触达后转化率、客单价。

Q2: Agent 的好坏怎么量化评估？

核心思路：分三层—— 1. 任务完成率（结果）：意图发起后有没有完成下单？类似搜索满足率 2. 过程质量（体验）：对话轮数、人工转接率、用户纠错次数 3. 商业指标（价值）：客单价、复购率、LTV

完成率判定靠隐式信号（下单/退货）+ 显式反馈（抽样 10% 弹👍👎）。早期人工标注建基准线，规模化后隐式为主。

Q3: Agent 在电商场景落地最大的风险？怎么控制？

核心思路：最大风险是自主决策出错导致信任崩塌。搜索给错用户划走就行，Agent 买错了是真金白银。

做三级权限：低风险全自动（查价格）→ 中风险确认后执行（下单）→ 高风险禁止自动（退款）。随信任积累逐步放权。原则：宁可多问一步，不可自作主张。

Q4: 你的情报系统能力怎么迁移到搜推广场景？

核心思路：情报系统的核心是从噪音中提取 signal。搜推广的核心问题一样——从用户模糊的搜索词中提取真实意图。

多源信号（搜索词 + 历史行为 + 用户画像）→ 意图推理 → 个性化结果。比如搜”健康零食”，结合用户上周买蛋白粉的行为 → 推蛋白棒而不是普通零食。

Q5: 加上 Agent 能力后，搜索推荐最大的变化是什么？

核心思路：从单轮交互变成多轮对话。搜索不再是独立模块，而是对话的第一轮。推荐不再是被动展示，而是 Agent 在对话中主动穿插关联推荐。

新增指标：对话深度（几轮完成购买）、跨品类渗透率（有没有带动关联购买）。

Q6: Agent 和 RPA 的本质区别？

核心思路：RPA 是”录制→回放”，流程固定、遇到意外就崩。Agent 是”给目标→自己想办法”，能理解、推理、应变。

类比：RPA = 流水线工人（按 SOP）、Agent = 实习生（给目标自己想办法）。不是替代关系，是各管各的层——确定性重复操作用 RPA，需要判断的用 Agent。

Q7: AI PM 和传统 PM 最大的区别？你为什么适合？

核心思路：最大区别是不确定性——传统产品行为确定，AI 产品行为概率性。

三个能力差异：评估方式（bad case 监控 vs 漏斗）、跟技术沟通（理解模型能力边界）、产品边界模糊（从意外用法发现机会）。

我适合因为我是 AI 产品的深度用户和构建者——搭过 Agent 体系、写过架构研报、每天在用在踩坑在迭代。

Q8: Agent 的记忆跟现有用户画像有什么区别？

核心思路：用户画像是结构化标签（行为数据被动采集，T+1 批量更新），Agent 记忆是非结构化上下文（对话中用户主动表达，实时生效）。

画像告诉你”该用户常买母婴产品”，Agent 记忆告诉你”宝宝 8 个月，刚开始吃辅食，对鸡蛋过敏”。两者互补，不是替代。Agent 获取的信息还可以回写到画像系统。

Q9: 入职第一天 leader 说”搜索结果页加个 AI 助手入口，你来负责”，第一周做什么？

核心思路：不急着画原型。Day 1-2 了解现状（跑一遍搜索链路、看数据看板、找”搜了但没买”的 case）。Day 3-4 定义问题（流失是因为”找不到”还是”选不了”？）。Day 5 输出 MVP 提案（先选一个垂直场景验证，定义 AB 对比指标）。

原则：第一周的产出不是方案，是问题定义。

Q10: 天猫超市 AI 助手和淘宝”问问”有什么区别？为什么不直接复用？（追问题）

核心思路：淘宝问问解决”不知道买什么”（探索性、多轮发散），天猫超市解决”帮我快点买完”（效率导向、越短越好）。

直接复用的问题：用户说”买牛奶”，淘宝问问会问”什么口味？全脂脱脂？“，但天猫超市用户期望的是”你上次买的 XX 牌还有货，比上次便宜 2 块，要下单吗？”

底层模型可以复用，但产品层的意图理解、对话策略、交互形态需要重新设计。

Q11: 天猫超市也在做非标品（生鲜、预制菜），Agent 策略跟标品有什么不同？

核心思路：非标品不确定性高（今天有什么、新不新鲜、口味合不合适），Agent 从”管家型”转向”顾问型”。

四个策略差异：①主动推荐替代品（缺货时不只说缺货）②时效感知（结合配送时间判断生鲜能不能送到）③场景化推荐（“今晚做什么菜”而不是”你该补货了”）④信任建设更难（需要更多解释降低决策焦虑）。

核心指标也不同：标品看复购率，非标品看品类渗透率和退货/差评率。

Q12: Agent 工具调用失败了怎么办？怎么设计容错？

思考框架：按严重程度分三层，每层对应不同处理策略。

“这个问题我在自己搭 Agent 的时候踩过坑。核心原则是：对用户透明，不要假装没事。

我会设计三层容错：

第一层：重试 + 降级 - 网络抖动导致的失败 → 自动重试 1-2 次，用户无感知 - 某个服务挂了 → 降级到备选方案。比如实时库存接口挂了，用缓存库存（可能不是最新但比没有好），同时告诉用户’库存信息可能有几分钟延迟’ - 价格接口挂了 → 展示上次缓存的价格，标注’价格更新于 X 分钟前’ - 关键是：降级不是沉默，要告诉用户你在用的是降级数据

第二层：局部失败不阻断整体 - 用户说’帮我买牛奶和纸巾’，牛奶下单成功但纸巾缺货 → 不要整个任务回滚，告诉用户’牛奶已下单，纸巾暂时缺货，要换一个品牌还是到货提醒你？’ - 把大任务拆成独立子步骤，每一步单独成功或失败 - 成功的保留，失败的单独处理

第三层：诚实兜底 - 真的搞不定 → 直接告诉用户’这个我暂时处理不了，帮你转到人工客服’，同时把对话上下文一起传过去，用户不用重新描述问题 - 最怕的是 Agent 假装成功实际没办成，或者卡在那里不说话

我在 intern-clawd 里也是这么设计的——Agent 遇到权限不够或工具出错时，不会自己硬上，而是告诉我’这个我做不了，原因是 XX，你可以选择 A 或 B’。这跟电商场景的容错思路是一样的：宁可坦诚说做不到，不可默默出错。”

Q13: Agent 会不会取代传统搜推系统？两者什么关系？

核心思路：不会取代，分层共存。搜推是基座（大规模、高并发、低成本、100%用户），Agent 是增强层（复杂意图、多轮决策、高成本、10-20%用户）。

类比：搜推是超市货架（负责广度），Agent 是导购员（负责深度）。Agent 不应该是独立产品，而是嵌入现有搜推流程的增强能力——搜索结果页加”帮我挑”入口，推荐流加”为什么推这个”交互。

Q14: 搜推系统会被 Agent 取代吗？（深度版）

思考框架：时间维度（短中长期）+ 功能层级（前台产品 vs 后台基础设施）+ 对淘天战略的理解

“我认为搜推不会被取代，但它的角色会发生根本性的变化——从前台产品退到后台引擎。

短期来看（1-2 年），搜推和 Agent 是分层共存的关系。大多数用户还是主要用搜索框，Agent 只服务于有复杂需求的用户群体。

中期来看（3-5 年），Agent 会成为主要交互入口，但用户跟 Agent 说的每一句话，Agent 在背后都在调用搜索接口、推荐接口、比价接口。搜推从”用户直接操作的产品”变成”Agent 调用的工具集”。用户看不到搜索框了，但搜推的能力没有消失。

长期来看，真正的竞争点会从”谁的搜推算法好”变成”谁的搜推能力更容易被 Agent 调用、接口更丰富、数据更精准”。

淘天正在做的”从流量分发到理解系统”——本质上是在为这个转型做基座准备。理解系统不是为了给人看，是为了给 Agent 调用。这是我理解的为什么这个方向有战略意义。”

Q15: Agent 时代搜推的新模式？

思考框架：数据归属权变化 + 机会与威胁两面看 + 结合自己项目举例

“我认为 Agent 时代最大的模式转变是推荐权力的转移——从平台侧中心化，转向用户侧去中心化。

传统搜推是这样的：你的画像存在平台服务器上，推荐算法是黑盒，所有数据归平台所有，用户是被推荐的对象。

Agent 时代是这样的：用户的偏好存在自己的设备上（类比我做的情报系统里的 MEMORY.md）、推荐逻辑对用户透明可解释、用户的 Agent 主动向平台拉数据而不是被动接收推送。

这对天猫超市是双刃剑。机会：率先把商品搜推能力以开放接口的形式提供给用户 Agent，成为 Agent 时代最好用的商品数据源，就像谷歌在 PC 时代成为网页数据源一样。威胁：用户 Agent 如果能跨平台比价——同时调天猫、京东、盒马——那平台的锁定力会显著下降，竞争从用户粘性变成接口竞争。

所以天猫超市现在该做的，是用 Context 积累建立护城河——用户在平台积累的购买记录、地址、偏好、家庭结构越多，Agent 在这里的表现就越好，迁移成本越高。”

Q16: 你的项目都是一个人做的，怎么证明跨团队协作能力？

思考框架：诚实 + 举具体协作经历 + 提炼 PM 思维（资源获取、需求倾听、自主验证）

“这是一个好问题，我不会虚报。确实我的项目大多是个人构建，但跨团队协作发生在另一个层面。

第一，需求对接和反馈迭代。情报系统的用户是公司技术高管，不是自嗨产品。我根据他们的反馈做了几次重要改动——加了 benchmark collector 模块（他们想追踪具体模型排名变化），把 webhook 纯文字推送改成文档形式（高管希望可以存档和分享）。改文档形式的过程中卡在内部文档 API 权限上——API 申请走了两周没批，我自己绕道用 Lobi（内部 OpenClaw 集成）实现了替代方案，没有让需求烂掉。这是典型的 PM 处理依赖阻塞的方式：找替代路径，不等着。

第二，跟外部 PM 配合。我参与过跟端到端 coding agent 的 PM 一起做技术选型讨论和约外部访谈。技术选型上我提供了对架构差异的分析视角，访谈上我帮助整理了问题框架。

第三，自己垫 API 费。情报系统 API 成本自己承担，先跑出效果再争取资源——这不是因为没有申请资源的渠道，而是 PM 思维：用最小成本证明价值比申请资源更快。这在公司内推项目的逻辑是一样的。

如果入职，我期望和有执行力的工程师团队配合——我的优势是需求定义清晰、技术沟通零成本、自己能验原型，对工程师来说沟通成本低。”

Q17: 情报系统的 LLM 筛选逻辑？怎么保证不漏重要信息？

思考框架：产品决策（precision vs recall 权衡）+ 工程实现（prompt 结构）+ 诚实的局限性认知

“我先解释一下系统的逻辑，再说我们主动选择了什么权衡。

筛选逻辑：不是让 LLM ‘凭感觉选’，prompt 里有明确的评分框架——我定义了 8 个视角维度（模型能力突破、政策监管动向、投资并购、学术突破等），以及 signal 强度判断标准：是不是 first（业内首次）、影响面有多大、时效性强不强、信源权威度如何。LLM 按这个框架打分，然后选出得分最高的 10 条。本质上是用 prompt 编码了一个评分函数。

防漏机制：多源覆盖——RSS 订阅了 40+ 信源，不同信源有交叉，同一条大新闻理论上会被多个信源捕获。同时每 4 轮 prompt 调优会 review 有没有漏掉重要事件，漏了就回头看为什么，然后修 prompt。

主动的权衡：我坦诚说，10 条一定会漏——我选择了高 precision 而不是高 recall。原因是用户是忙碌的技术决策者，他们的痛点是信息太多看不完，而不是信息太少。所以宁可漏掉一些，也不能用噪音淹没 signal。这个产品决策我是主动做的，不是被动的。

这个逻辑迁移到天猫超市的搜推场景——推荐排序也是 precision 优先，推 3 个精准的比推 30 个凑数的价值更高。”

Q18: 反问面试官

思考框架：问题要展示思考深度，而不是问”有什么发展空间”这种套话。好的反问让面试官记住你。

推荐问 2 个（按面试剩余时间选）：

问题 1（技术/产品深度）： “天猫超市目前在 Agent 方向最大的技术或产品卡点是什么？是意图理解准确率不够，还是多轮对话的上下文管理，还是跟现有搜推系统的集成？”

为什么这么问：展示你对 Agent 落地挑战有具体认知，不是泛泛谈概念。面试官的回答也能帮你了解实际工作内容。

问题 2（岗位期望）： “这个岗位入职后前三个月，您期望实习生能交付什么？是更偏调研分析，还是有机会参与实际产品需求定义？”

为什么这么问：展示你以结果为导向，来了就想有实际产出，不是混经历。

备用问题（如果已聊过很多技术细节）： “您觉得这个团队未来一年最大的变量是什么——是技术成熟度，还是业务场景的拓展速度，还是组织协调？”

Q19: GMV 突然下降 30%，怎么拆解原因？

思考框架：数据验真 → 公式拆解 → 维度下钻 → 假设排查 → 输出结论

“我会分四步走：

第一步，确认数据真实性——30% 的降幅很大，先问三个问题：口径是不是变了？是不是赶上了大促后的自然回落（基数问题）？有没有数据系统 bug 或埋点错误？这一步不是质疑数据，是避免在错误的数据上浪费分析时间。

第二步，沿公式拆解——GMV = 访问用户数 × 转化率 × 客单价。三个因子先看哪个降了：用户数降说明流量问题（外部导流或 SEO 有变化）；转化率降说明产品或信任问题；客单价降说明结构性变化（用户在买低价商品）。

第三步，维度下钻——确认是全局问题还是局部问题：地区（只有某个城市？仓储问题？物流问题？）、品类（只有某个品类？竞对有补贴活动？）、用户群（只有新用户？还是老用户流失？）、时间（持续下降还是单天异常？）。

第四步，假设排查——结合外部信息：竞对有没有大促？平台算法有没有调整？政策有没有变化？有没有舆情负面事件？

回答时我会先说：‘我的思路是先验真实性，再沿 GMV 公式逐层拆，同时看是全局还是局部。您希望我从哪个方向深入？’”

Q20: 怎么设计 AB 测试验证 AI 推荐效果？

思考框架：假设定义 → 分组设计 → 指标体系（主/辅/护栏）→ 实验规范 → 常见陷阱

“我会按五步来：

第一步，定义可证伪的假设——不是’AI 推荐更好’，而是’AI 推荐会使 7 天转化率提升 X% 以上，且退货率不上升’。假设要具体、可量化、有方向性。

第二步，分组设计——按用户 ID hash 随机分（不按时间段，避免时间混淆）。先 5% 小流量跑 3 天看有没有灾难性问题，再放大到 50% 跑完整周期。对照组保持原版搜推，实验组开 AI 推荐。

第三步，指标分三层——主指标只选一个（搜索转化率），防止多重检验问题；辅助指标看方向（客单价、加购率、对话深度）；护栏指标是红线（退货率、客诉率、页面报错率——这三个变差就暂停实验）。

第四步，实验规范——样本量提前用统计功效算好（效果量假设 5%，置信度 95%，至少需要多少用户）。至少跑 7 天覆盖完整周内周期，避开大促（大促行为失真）。

第五步，避坑——新鲜感效应：第 1 周数据可能虚高，要看第 2 周是否稳定；辛普森悖论：全局好不等于分层好，一定要分用户类型、品类看；多重检验：主指标只定一个，不能后验选最好看的指标汇报。”

Q21: 上线前怎么做离线评估？

思考框架：自动化指标（广度）+ 人工 case review（深度）+ Agent 特殊评估方法 + 局限性认知

“离线评估我分三层做：

第一层，自动化回测指标——用历史数据跑，看三类指标：Recall@K（推的前 K 个里有没有用户实际买的商品，衡量召回质量）；NDCG（排序是不是把对的排前面）；覆盖率/多样性（是不是千人千面，还是所有人都看到同样的东西）。这一层跑得快，能快速发现大方向问题。

第二层，人工 case review——抽 100-200 个代表性场景，人工标注合理/不合理/有风险。天猫超市特别需要关注的 bad case：推了用户曾经标记过敏的食品（安全风险）；推了刚买完还没用完的商品（浪费用户注意力）；推了库存紧张或已下架的商品（点进去买不了，体验差）；推荐和用户家庭结构不匹配（单身用户推婴儿食品）。这一层保证没有低级错误。

第三层，Agent 特殊评估——纯统计指标评估不了对话质量。用历史真实对话 log 回放：给 Agent 同样的输入，看它的输出和历史人工客服的对比。再用 LLM-as-judge（让另一个大模型按标准打分）批量评估。

局限性要诚实：离线评估保下限（避免上线出大问题），上限靠在线 AB 验证。离线永远有分布偏差——历史数据代表过去的用户行为，新的 AI 交互会产生历史上没有的行为模式。”

Q22: 怎么快速判断一个 AI 功能是否值得上线？

思考框架：价值假设验证 + 成本估算 + 风险评估 + 决策框架

“我会从三个维度快速判断：

第一，价值假设是否成立——核心问题是：这个 AI 功能解决的是用户的真实痛点，还是技术可以做但用户不需要？快速验证方法：找 10-20 个真实用户做 5 分钟访谈，或者用现有数据看有没有对应痛点的信号（比如某个场景的搜索零结果率高、转化率低、用户反复回来搜）。

第二，成本和可行性——AI 功能的成本结构特殊：训练成本（一次性）、推理成本（按请求量线性扩张）、标注和维护成本（持续投入）。要估算：按日活用户量，每天的 token 消耗成本是多少？跟收益做粗略对比。

第三，最大风险——AI 功能的风险不是普通 bug，是’表现不稳定导致信任崩塌’。上线前问：如果出现最坏情况（推错了商品、答错了问题），用户损失多少？平台损失多少？有没有 fallback 方案？

判断标准：价值假设清晰 + 成本可接受 + 风险可控 → 上线；任一不满足 → 先做原型验证再决定。

我在情报系统上也用了这个逻辑——先用最小成本（自己垫 API 费）验证高管真的会看，再决定是否投入更多资源做工程化。”

Q23: 新鲜感效应消退 vs 产品本身有问题，怎么区分？

思考框架：用户分群（时间维度）+ 使用深度（质量维度）+ 判断标准清晰化

“这是 AI 产品上线后最常见的混淆问题，关键是看两组数据：

第一组，用户时间分群——把用户按进入时间分成’第一周进入’和’第二周新进入’两组。如果第二周新进入的用户数据仍然好（转化率正常、对话深度正常），说明产品本身没问题，只是早期用户的新鲜感过了。如果第二周新用户数据也差，说明产品本身有问题。

第二组，使用深度而不只是频次——频次（DAU、打开次数）容易受新鲜感影响。更有信号的是使用深度：留下来的用户，对话轮数有没有增加？下单率有没有提升？用户有没有主动回来复用？如果留下来的用户质量在提升，说明健康的筛选效应，不是产品崩坏。

判断标准：第二周留存用户的人均 GMV 没降甚至涨了 = 健康消退。第二周留存用户的各项质量指标全面下滑 = 产品有问题。

结论之后的动作也不同：健康消退 → 优化用户激活（让更多用户坚持过新鲜感阶段）；产品有问题 → 找根因（bad case 分析，找哪类场景最多人退出）。”

Q24: AI 搜索怎么处理”好喝的牛奶”这种主观搜索词？

思考框架：主观词客观化（两条路径）+ 个性化分层 + 新用户冷启动

“主观搜索词的本质问题是：’好喝’对每个人的含义不同，不能用一个固定答案回复所有人。我会分两层处理：

第一层，群体共识——用评价数据挖掘出’好喝’这个标签在用户社区里的实际映射。比如分析天猫超市里牛奶商品评价中’好喝’出现频率最高的商品，以及这些商品的共同属性（全脂？某个品牌？某种风味？）。这给了一个群体共识的基线答案。

第二层，个人偏好覆盖——结合用户历史购买行为：用户一直买全脂牛奶 → ’好喝’对他大概率是全脂；用户最近在看低脂食品 → ’好喝’可能是低脂口感。用历史行为把主观词翻译成客观特征。

新用户冷启动——没有历史数据时，做一轮轻量澄清：’您是偏好哪种口味？纯正奶香还是淡一点的？’一个问题拿到一个信号，不要问多。

结果呈现——不要只给一个结果，给 2-3 个理由各异的选项：‘这款是口碑最好的（评价里好喝提及最多）；这款是根据您的历史偏好推的；这款是今天特惠款。’ 让用户感受到 AI 在理解他，而不是随机推荐。”

Q25: 怎么验证”兴趣推理”真的比”行为相似”好？

思考框架：先解释本质区别 → 再说验证指标设计 → 引用数据支撑

“先说两者的本质区别：

行为相似是’买 A 的人也买了 B’——纯统计关联，不知道为什么，也不知道用户的真实状态。它的上限是历史行为的延伸，推不出用户还不知道自己需要的东西。

兴趣推理是语义理解：‘你买了奶粉 + 看了辅食教程 + 搜了 8 个月宝宝 → AI 推断你家有 8 个月宝宝在过渡辅食阶段 → 推米粉、磨牙棒、辅食工具’。这是主动的状态推断，不是被动的行为匹配。

验证方法——AB 测试看两个核心指标：

第一个：点击率和转化率（基础）——兴趣推理应该不比行为相似差，如果差了，说明推理出错了。

第二个：惊喜度（区别度指标）——定义为’推的商品里有多少是用户从未浏览过的品类但最终购买了’。惊喜度高说明兴趣推理真的帮用户发现了他自己没想到的需求，这是行为相似做不到的。

判断标准：惊喜度高且转化不差 = 兴趣推理真的在工作。淘天数据：推荐点击 +10%、购买效率 +25%，这个幅度说明确实有超出行为相似的增量价值。”

Q26: AI 广告动态出价，商家怎么信任你？

思考框架：信任三要素（透明、保障、渐进）+ 从商家视角看决策动机

“商家不信任 AI 出价，根本原因是’AI 花的是我的真金白银，出了问题我亏损’。建立信任要从三个维度入手：

第一，透明度——让商家看到 AI 在做什么决策，而不是黑盒。每次出价调整都附说明：’今天下午 3-5 点竞争度低，我把出价降了 15%；晚上 7-9 点转化高，我提了 20%。’商家看得到逻辑，才能判断是不是合理，才愿意信任。

第二，保底机制——商家设定 ROI 下限（比如我的 ROI 不能低于 3），AI 在这个约束内优化。低于下限自动暂停，而不是继续花钱。这把商家的风险敞口限定在可接受范围。

第三，渐进信任——不要一开始就让商家把全部预算交给 AI。第一周 10% 预算跑 AI，90% 手动；看一周数据；AI 表现好就逐步放权。小预算先拿到数据，数据说话比任何说辞都有力。

个性化效果报告比行业平均值重要——别跟商家说’行业平均 ROI 提升了 20%‘，要说’您这个店铺上周 AI 出价的 ROI 是 X，手动出价是 Y，差异是 Z’。自己的数据比行业数据更有说服力。”

Q27: AI 美工 2 亿张图怎么保证质量？

思考框架：质量分层（自动化下限 + 人工上限 + 用户兜底）+ 不同风险等级的不同处理

“2 亿张图不可能全靠人工审，但也不能完全信任自动化。我会做三层质检架构：

第一层，自动化门控（100% 覆盖）——所有生成的图都过自动化检测：品牌一致性（logo 没变形、颜色符合品牌规范）、违规检测（无违禁词、无敏感图像、合规）、基础质量（分辨率达标、不模糊、主体完整）。这一层是下限，把明显的错误全部拦掉。

第二层，人工抽样（1-5%，风险加权）——不是随机抽，是风险加权抽：新品牌第一批图重点看（还没建立自动化质量基准）；新品类第一批图重点看（模型对这个品类的理解可能不准）；评分低的自动化标注需要人工复核。人工保上限，抓系统性问题。

第三层，商家兜底（最终决定权）——所有生成图上线前商家可以预览，一键修改或回滚到上一版本。这是最重要的安全网——商家最了解自己品牌调性，最终决定权给商家，出了问题可以即时撤回。

原则：自动化保下限（拦灾难性问题）、人工保上限（提升系统性质量）、商家有最终决定权（保信任）。随着数据积累，自动化门控的标准会越来越精准，人工抽样比例可以逐步降低。”

Q28: AI 客服省 2000 万但”听不懂”，怎么平衡成本和体验？

思考框架：问题分层 + 指标重定义 + 转人工不是失败

“这道题的关键是：不要把’AI 拦截率’当成成功指标——那会导致产品方向错误（想方设法不让用户转人工，而不是真正解决用户问题）。

正确的分层服务逻辑：

简单、确定性问题全自动：查物流状态、退换货政策说明、优惠券使用规则——这类问题答案唯一、无歧义，AI 准确率可以做到 95% 以上，且用户接受 AI 回答（反正答案都一样）。这才是省 2000 万的来源。

复杂、判断性问题及时转人工：买了坏了要维权、一单多件漏发了几件、用户情绪明显激动——这类问题 AI 理解错了代价很高，应该快速识别并附带完整对话上下文转人工。转人工不是失败，是正确分工。

关键指标重定义：不看 AI 拦截率，看三个指标——首次解决率（第一次回复就解决了问题，不用再问第二次）；转人工后满意度（转接体验顺不顺，上下文有没有带过去）；重复来电率（同一个问题用户来了几次——重复多说明根本没解决）。

一句话原则：省钱不是靠堵死转人工的路，是靠让 AI 真正把它擅长的 80% 解决好，然后把剩下的 20% 优雅地交给人工。”

Q29: AI 帮我挑 vs 搜索的本质区别？产品形态？

思考框架：决策权分工 → 用户心理模型 → 产品设计细节

“两者的本质区别是决策权的分工：

搜索 = 用户主导决策，AI 只负责找。用户说’搜牛奶’，搜索把 200 个结果排给你，你自己选。AI 是工具，决策在用户手里。

帮我挑 = 用户主动让渡决策权，AI 负责找+选。用户说’帮我挑一款牛奶’，隐含的意思是’你帮我做选择，给我一个或几个答案就行’。这需要 AI 理解用户的隐性偏好和当前情境，不是列清单，是给建议。

用户心理模型也不同：搜索用户是’我知道自己要什么，我来找’；帮我挑用户是’我懒得想了，AI 比我更懂’。后者对 AI 能力的信任要求更高，一旦推错了信任成本也更高。

产品形态设计：

入口：放在搜索结果页顶部（用户搜了但没有直接点购买的时候，说明在犹豫，这时候最适合接手）；也可以放在品类页无结果或结果过多时触发。

对话设计：最多 2-3 轮——超过 3 轮用户就不耐烦了。第一轮澄清最核心的约束（给谁买？预算多少？），第二轮确认，第三轮给结果。

输出：给 2-3 个选项而不是 1 个（用户需要一点控制感），每个选项附一句推荐理由（‘这款是口碑最好的，适合追求品质的用户’）。

关键细节：直接可加购，不跳详情页——用户选了就是信任了，还要他再看一遍详情页是多此一举，会流失。”

Q30: AI 试穿退货率反而上升了怎么办？

思考框架：诊断根因（期望管理问题）→ 解决方向（不是关掉功能）→ 指标重定义

“退货率上升第一反应不是关掉功能，而是先诊断根因。

根因分析：AI 试穿退货率上升，最可能的原因是效果’太好’了——虚拟试穿效果过于完美，让用户产生了超出实物的期望。用户看到的是 PS 级效果，收到的是真实商品，落差产生退货。这不是功能失效，是期望管理失败。

解决方向——管理预期，不是限制功能：

第一，真实感优先：渲染引擎参数调整，保留适度的皱褶感、阴影不均匀感，不追求’完美’而追求’接近真实’。

第二，加买家秀对比：在 AI 试穿图旁边并排展示真实买家秀，让用户能对比虚拟和真实的差距，主动降低期望。

第三，标注置信度（最重要）：不同商品类型的 AI 试穿准确率差异很大——纯色基础款 T 恤准确率很高（布料结构简单），蕾丝/复杂印花/立体结构准确率低。在商品上标注’AI 试穿效果参考度：高/中/低’，低参考度的商品加备注’实物与效果图可能有差异，建议参考买家秀’。

指标重定义——核心指标不是 AI 试穿使用率，而是’试穿后购买且未退货率’。使用率高但退货率也高 = 功能在伤害平台。使用率适中但购买未退货率高 = 功能在创造真实价值。追踪这个指标驱动优化方向。”

Q31: 智能购物车凑单功能的成功指标？

思考框架：双赢目标（用户 + 平台）+ 三层指标体系 + 防虚假繁荣的护栏指标

“凑单功能的设计目标是双赢——用户凑满免运费或满减省了钱，平台客单价提升了。这两个目标必须同时成立，任何一个失败都不算成功。

用户侧指标： - 凑单点击率（推荐出来的凑单商品有多少人点了） - 用户实际节省金额（满减凑成了多少） - 购物车放弃率变化（凑单推荐出现后，放弃购物车的比例是升还是降——如果升了说明推荐打扰了用户）

平台侧指标： - 客单价提升（加了凑单商品后，订单均价提升了多少） - 凑单商品的毛利率（不能只看 GMV，要看凑单商品是不是利润贡献正向的） - 订单完成率（有没有因为凑单商品问题导致整单取消）

护栏指标（红线）： - 凑单商品退货率（凑单买的东西不需要，退货率会比自然购买高） - 凑单后取消率（用户加了凑单商品又删掉，说明推荐不准）

主指标：凑单带来的增量 GMV × (1 - 凑单商品退货率)。乘退货率是防虚假繁荣——凑单 GMV 高但退货率也高，实际是在搬砖，平台并没有真正增收，用户也没有真正得到价值。”

Q32: 你选模型的标准是什么？不同场景怎么选？

思考框架：四维评估框架 + 自己的实践经验 + 具体场景映射

“选模型我有四个维度，按优先级排：

第一，任务匹配度——不看 benchmark 排行榜，自己跑真实业务 case 看效果。Benchmark 是通用任务，业务场景非常具体，排名第一的模型在你的任务上不一定最好。跑 20-50 个代表性 case，看输出质量，这才是真实判断。

第二，成本——按实际用量估算月成本。特别是推理成本：输入 token 多少钱、输出 token 多少钱、按 QPS 估总量。成本在某些场景会成为主要约束。

第三，延迟——用户面向的实时功能，首 token 延迟和总响应时间是核心。AI 客服回复超过 3 秒用户就不满意了。离线批处理任务延迟不重要。

第四，稳定性——有没有 API 稳定性 SLA？历史宕机记录？对于生产级产品，不稳定的模型服务比贵一点的稳定服务代价更大。

实际经验：情报系统选 DeepSeek——离线运行、低请求量、性价比高，满足需求。如果是天猫超市的 AI 客服，我会用分层策略：简单确定性问题（查物流、查政策）用小模型（快+便宜，延迟低）；复杂判断性问题（纠纷处理、多步骤退货）用大模型兜底（准+贵，但用量少）。不是所有问题都需要 Opus 来回答’快递在哪了’。”

Q33: AI PM 需要会写代码吗？技术背景是优势还是包袱？

思考框架：优势清单 + 包袱诚实承认 + 我是怎么做的

“我认为技术背景对 AI PM 是净正向，但要主动管理它的副作用。

三个核心优势：

第一，需求可行性判断不依赖开发——当工程师说’这个做不到’或’需要三个月’，我能自己判断是真的技术难题，还是实现路径选得不对。这减少了来回确认的时间，也减少了被糊弄的可能性。

第二，跟工程师沟通零翻译成本——我说’这个接口的 latency 加了 200ms 对用户体验的影响比功能本身大，优先级能不能调整一下’，工程师听得懂，不需要翻译成业务语言再翻回来。

第三，能自己写原型快速验证假设——情报系统我自己从 RSS 采集到 LLM 分析到 webhook 推送全栈搭出来。能在 3 天内验证一个想法值不值得投入，而不是等开发资源排期 3 周。

诚实的包袱：容易陷入技术细节，忘了用户视角。上一轮面试官给的反馈’思路打开一些’，就是这个问题的直接体现——我太快讲技术实现，没有先讲用户痛点和业务价值。这是我主动在改的习惯：每次回答问题，先说用户场景和业务目标，技术方案放后面。

一句话总结：技术是看问题的放大镜，但决策的准心要始终对准用户价值，不是对准实现的优雅度。”

Q34: 天猫超市 vs 竞品（京东/美团/盒马/山姆）

思考框架：各平台差异化定位 → 天猫壁垒 → 学竞品什么 → Agent 在各平台的价值差异

“我从定位差异、天猫壁垒、和 Agent 视角三个维度来说：

各平台核心定位： - 天猫超市：SKU 全 + 88VIP 生态 + 淘宝流量入口 - 京东：物流极快 + 品质感（自营保证） - 美团买菜：30 分钟极速 + 生鲜强 - 盒马：新零售 + 自有品牌 + 线下体验 - 山姆：会员制 + 极致精选（SKU 少但每个都是精品）

天猫的核心壁垒：第一，淘宝主站流量——用户搜’牛奶’直接命中天猫超市商品，不需要下载另一个 app，流量成本天然低。第二，88VIP 跨生态打通——电商+视频+本地生活一张会员卡，用户粘性不只在购物。第三，行为数据量最大——全国最大电商平台的购买数据训练搜推模型，学习样本量竞对无法追上。

该学竞品什么：学美团的即时性——天猫超市的次日达在生鲜和日用快消场景有时慢了，学美团 1 小时达是补短板；学山姆的选品逻辑——山姆 SKU 少但’买就对了’的信任感极强，这恰好是 AI 帮我挑可以实现的——用 AI 代替人工精选，解决天猫 SKU 太多带来的决策负担。

Agent 在不同平台的价值定位不同：天猫/京东 SKU 多 → Agent 帮你选（决策辅助是核心）；美团/盒马即时性强 → Agent 帮你快（一句话下单，减少操作步骤）；山姆客单价高 → Agent 帮你省（最佳补货时机、价格预警）。同样是 Agent，产品设计要贴合平台的核心用户心智。”

Q35: 用 AI 缩短天猫超市的下单链路？

思考框架：场景分类（复购 vs 新购）+ 步骤压缩设计 + 关键设计原则

“下单链路优化要先分场景，因为复购和新购的用户心理模型完全不同。

复购场景（占天猫超市大头）：用户心智是’快速补货’。现有链路：打开 app → 搜索 → 筛选 → 找上次买的 → 确认规格 → 加入购物车 → 结算 → 支付，大概 8 个步骤。AI 可以压到 2 步：用户说’上次那个牛奶再来一箱’ → AI 匹配历史购买记录确认商品规格 → 确认支付。甚至可以做到 0 输入——AI 主动提醒：‘您的光明牛奶按购买频率预计快用完了，要补货吗？一键下单。’

新购场景：用户需要决策，不能强行跳过，但可以大幅缩短。原来是’面对 200 个搜索结果自己选’（认知负担极高）。AI 变成：理解需求 → 直接给 3 个带推荐理由的选项 → 直接加购，从 200 选 1 压到 3 选 1。

关键设计原则：步数少不是绝对好，关键是让用户可以跳过，而不是替用户跳过。用户信任高的场景（熟悉商品、小额、复购）可以 AI 主动，用户信任低的场景（新商品、大额、第一次买）要把决策权还给用户。

举例：AI 主动说’帮您重复下单上次的牛奶，15 秒内可取消’，这是让用户可以跳过。AI 直接下单不告知，这是替用户跳过——这是错误的，会建立信任危机。”

Q36: prompt 调优有哪些技巧？

思考框架：结合自己情报系统实践 + 7 条方法论 + 说清楚每条为什么管用

“我在情报系统做了 4 轮 prompt 调优，总结出 7 条实战方法论：

①角色定义要具体——不说’你是一个 AI 助手’，而是说’你是一个服务 25-35 岁精打细算妈妈群体的天猫超市导购，对用户推荐时优先考虑安全性和性价比’。角色越具体，模型的输出风格和判断标准越一致。

②给 few-shot 示例而不是给规则——规则是’推荐时要说明推荐理由’，示例是直接给 3 条带理由的推荐范例。模型理解示例的能力比理解抽象规则强得多。2-3 个好示例比 10 条规则更管用。

③先解决方向错误再优化细节——如果推荐方向完全跑偏（推了不相关品类），先改大方向；不要还没解决方向问题就去调输出格式。调优顺序：意图理解 → 内容质量 → 格式输出。

④约束输出格式——用 JSON 或固定 markdown 结构，而不是自由文本。让下游系统好解析，也让模型输出更稳定（不会一会儿有推荐理由、一会儿没有）。

⑤分步拆任务——复杂任务拆成多个 prompt：第一步筛选相关商品，第二步对选出的商品做排序和理由撰写。一步做两件事会降低精度。

⑥用 bad case 驱动迭代——每轮调优不是凭感觉改，是收集具体的 bad case（这条推荐为什么不对），分析根因，针对性改 prompt。情报系统 4 轮调优都是 bad case 驱动的。

⑦温度/top_p 是最后才调的——很多人一上来就调 temperature，实际上 prompt 本身没有调好的情况下，调超参数是在做无用功。先把 prompt 调好，最后才微调温度。”

Q37: AI 客服上线后满意度降了但省了成本，怎么汇报？

思考框架：先诊断再汇报（不能拿着坏数据直接汇报）+ 汇报结构（现状→根因→建议→保留价值）

“遇到这种情况，我不会直接把数据扔给老板——先自己把根因搞清楚，带着诊断和建议去汇报，而不是带着问题去让老板做判断。

第一步，拆根因：

按问题类型拆：简单问题（查物流、退换货）的满意度有没有降？复杂问题（投诉、纠纷）的满意度有没有降？如果只有复杂问题降了，说明分流策略有问题，AI 承接了它处理不了的问题。

按失败模式分类：收集 bad case，看失败是因为’答错了’（能力不足）、‘没听懂’（意图理解失败）、还是’态度冷’（交互设计问题，用户感受不到温度）。三种根因对应三种解法。

对比基准：和上线前人工客服处理同类问题的满意度对比，确认降了多少是 AI 带来的，多少是整体客诉环境的变化。

第二步，做决策：AI 擅长的问题（简单确定性、满意度没降）继续；AI 不擅长的问题（复杂判断性、满意度明显降）直接转人工处理，不强行 AI。

汇报结构：‘上线后整体满意度降了 X%，但拆分来看——简单问题（占 70% 量）满意度持平，是 AI 擅长的场景，每月省成本 XX 万，建议继续。复杂问题（占 30%）满意度降了 Y%，根因是 AI 被迫处理了投诉类场景，建议这类问题直接转人工，预计恢复满意度 Z%，成本多 XX 万但体验上可以接受。净结果：每月还能省 XX 万，同时满意度恢复。’”

Q38: 情报系统怎么衡量用户是否真的在看？

思考框架：用真实经历回答 + 展示数据驱动迭代的 PM 思维

“这是情报系统里我做的一个真实改造，我把完整过程讲一下：

初始问题：系统上线初期用 webhook 推送纯文字消息到群里，完全没法知道有没有人看——消息发出去就消失在信息流里了，高管有没有打开、看了多久，完全是黑盒。

改造方向：把推送形式从纯文字消息改成自动生成并发送文档（飞书/内部文档）。文档有原生浏览记录——谁打开了、什么时候打开的、在文档里停留了多久、看到哪里——这些数据平台自动记录，不需要额外埋点。

执行过程中遇到阻塞：内部文档 API 权限申请走了审批流，等了两周没批下来。我没有干等，绕道用 Lobi（内部 OpenClaw 的集成）实现了同等功能——能自动发文档并获取浏览记录，权限走的是 Lobi 的授权，不依赖单独申请。

改完之后发现了什么：数据显示某几位高管每天都准时打开，看完整篇；另外几位从不打开。这个发现直接驱动了内容调整——我去跟每天看的高管深访，理解他们觉得哪些类型的 signal 最有价值，用这个反馈来优化内容框架。

这个经历说明一个 PM 逻辑：没有数据就先把数据搞来，有了数据就让数据驱动迭代。”

Q39: 搜推广的未来——从面向人到面向 Agent（⭐主动抛出的亮点题）

思考框架：从自己项目引入 → 核心判断（交互范式转变）→ 平台的机会与挑战 → 天猫超市应该怎么做

“我在自己的项目里做了一个实验，让我对这个问题有了不同的视角。

我做了一个 AI 新闻推荐 skill：推荐逻辑是 Agent 读取用户本地的 USER.md 和 MEMORY.md（记录了用户的兴趣领域、工作背景、关注重点），然后从信源里筛选最相关的内容推荐。这套逻辑完全不依赖平台中心化推荐算法——Agent 在本地做个性化，平台提供内容数据源。

这让我意识到一个趋势：平台搜推广正在从面向人转变为面向 Agent。

现在的搜推靠什么影响用户？UI 设计、标题吸引眼球、图片视觉、营销文案。这些手段是专门为’人类的注意力’设计的。但如果用户的 Agent 替他们看，Agent 不吃标题党，它看的是结构化数据：价格准确吗？评分真实吗？库存有没有？跟用户偏好匹配度是多少？

平台搜推不会消失，但它的形态会变化——从’把商品展示给人’变成’给 Agent 提供结构化商品数据接口’。广告也一样，从抢用户眼球变成抢 Agent 的优先级排序。

天猫超市现在该做三件事：

第一，把隐性 Context 显性化——用户散落在各处的行为数据（买过什么、搜过什么、退过什么）转化成结构化的用户理解档案，让 Agent 能调用。

第二，用 AI 产品创造新的 Context——对话式购物产生了行为数据挖不出的信息，比如用户在对话里说’宝宝乳糖不耐受’，这种信息只有通过 Agent 对话才能获取。

第三，让 Context 不可迁移——用户在天猫超市积累的购买记录、偏好、家庭信息越多，他们的 Agent 在这里表现越好，迁移到其他平台 Agent 就会变差。微信靠社交关系锁用户，天猫超市要靠购物 Context 锁用户。

终局判断：未来的竞争力不是 SKU 多少和物流多快（这些都能被对手卷到差不多），而是你对这个用户理解有多深。每个用户在平台上积累的 Context 是独一无二的，这才是真正的护城河。”

Q40: 今天体验天猫超市 AI 导购的真实发现（⭐面试必用）

思考框架：结构化产品分析（功能结构 → 亮点 → bad case → 改进建议），展示你是真的用了、真的在想

“我上线前专门体验了一遍天猫超市的 AI 导购，分享几个真实发现：

产品结构：AI 导购有三个核心能力——查同款低价（帮用户比价，解决’买贵了’焦虑）；推荐商品（决策辅助）；查找功能（App 内导航，比如搜’退货在哪里办’直接跳转对应入口，这个挺好用）。

亮点：搜泡面时，导购没有直接列结果，而是问’您是不是想回购之前买过的 XX 牌泡面？’——用购买历史做意图预判，把原本可能需要 3 轮对话的筛选压缩成 1 轮确认。这个设计思路很对，节省了用户描述需求的成本。

我的改进建议：不要等用户来搜，而是主动出击。按购买频率预判’你的泡面可能快吃完了’，在合适的时间点主动推送提醒，这才是从被动响应到主动服务的升级。

Bad Case 1：推荐了下架商品——导购推荐了 5 款泡面，其中 2 款点进去已下架。根因：推荐模块和库存系统没打通。方案：输出前加一步库存过滤，缺货或下架商品不进入推荐候选集。这是产品和工程之间的系统对齐问题。

Bad Case 2：重复推荐被拒绝的商品——我第一次推荐没买，退出重新进入，推荐结果一模一样。根因：系统没有记住上一轮的负反馈，’看了但没买’是非常强的负信号，却完全没有被利用。这呼应我说的 Context 理论——不只要记买了什么，也要记拒绝了什么，拒绝信号有时比购买信号更有价值。

Bad Case 3：多轮对话触发反爬验证码——AI 导购本来就是多轮交互，请求频率比普通搜索高。但后端用的是跟普通搜索共用的反爬机制，没有针对 AI 对话场景做豁免，结果多问几轮就弹出验证码，完全打断了对话流。这是典型的新功能和老基础设施之间没有对齐的问题，需要工程侧识别 AI 导购的请求类型并做单独的限流策略。”

Q41: 面试备用——你怎么看 MCP（Model Context Protocol）对电商的影响？

思考框架：先解释 MCP 是什么 → 对电商行业影响 → 天猫超市的机会与威胁 → 应对策略

“MCP（Model Context Protocol）是 Anthropic 推出的一个开放协议，让 AI Agent 可以标准化地连接外部服务——数据库、API、工具，不再需要每个平台各自写不同的集成代码。类比 USB-C：有了统一接口，任何设备都可以连。

对电商的根本影响：有了 MCP，用户的 Agent 可以同时调天猫、京东、盒马的接口，帮用户跨平台搜索、比价、下单。用户不再需要在多个 app 之间切换，Agent 替他完成全部操作。

对天猫超市的威胁：

第一，平台竞争维度变了——从’谁的 app 体验好、UI 好看’变成’谁的 MCP 接口更开放、数据更丰富、API 更稳定’。以前的护城河（界面体验、搜索功能）在 Agent 时代价值大幅缩水。

第二，用户跨平台比价变得零摩擦——现在用户懒得换 app 比价，未来 Agent 一秒钟帮你比完三个平台的价格。平台之间的价格竞争会加剧。

对天猫超市的机会：

第一，率先开放高质量 MCP 接口，成为 Agent 时代最好用的商品数据源——就像谷歌在网页时代成为信息入口，天猫超市可以成为 Agent 的商品数据入口。

第二，用 Context 数据构建护城河——MCP 让接口标准化，但它迁移不走用户在天猫积累的购买历史、偏好数据、地址信息。你的购买记录都在天猫，Agent 在这里表现最好，迁移到其他平台 Agent 就要从零开始学习你的偏好。这是真正的迁移成本。

一句话总结：MCP 会让平台竞争更激烈，但会者多劳——谁先把自己的数据和服务开放给 Agent 生态，谁就先占到 Agent 时代的入口位置。”
