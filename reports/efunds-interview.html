💼 五、Mock 12 题（易方达专属）

A. 动机 / 定位（3 题）

Q1. 为什么选择基金行业而不是互联网 / 投资机构 / 券商？
公式：不贬低其他，只讲你的主动选择
  • 互联网 AI PM：优化已有产品，边际空间在缩小
  • 投资机构（VC）：跨行业视野好，但一级市场节奏不稳
  • 券商：交易驱动，AI 更偏量化
  • 基金（尤其易方达）：**长期价值导向 + 投研驱动 + AI 落地空间大**
  • 关键：基金行业的 AI 落地不是追热点，是真实业务痛点——研报处理、因子挖掘、合规审查这些痛点非常具体
  • 易方达的 EFundGPT 和 投研平台获央行奖 → 说明公司确实在系统性推进 AI，不是 PPT 工程
Q2. 你怎么理解"智能应用研究员"这个岗位？
公式：拆三层——研究（前沿追踪）+ 应用（场景转化）+ 研究员（不是 PM）
  • "研究"：跟踪 AI 学术 + 行业动态（你的 AFI 日常）
  • "应用"：把研究转化为业务解决方案（不是单纯写 paper）
  • "研究员"：不是做 UI / 不做 C 端功能，是做课题型产品
  • 和 PM 的差别：PM 是需求驱动 / 快速迭代；研究员是课题驱动 / 深度钻研
  • 核心动作：发现痛点 → 调研技术 → 搭 PoC → 推动落地 → 评估效果
Q3. 实习 3 个月你期望交付什么？
  • 第 1 个月：跑通现有业务流程 + 找 1-2 个最痛的点
  • 第 2 个月：出 MVP（选最小可验证场景）
  • 第 3 个月：用户反馈 + 迭代 + 沉淀方法论（prompt 规范 / 评估框架）
  • 原则：不追求 3 个月做成大事，追求把一个小事做透 + 可复用

B. AI + 基金场景（4 题 —— 重点）

Q4. 你觉得 AI 在基金行业最大的落地机会是什么？
公式：主推智能投研（AFI 同构）+ 具体场景 + 量化价值
  • 首选：智能投研——基金公司每天处理巨量研报、新闻、公告、财报，研究员 80% 时间在信息处理，20% 在判断。AI 能把这个比例反过来
  • AFI 挂钩：我做的就是信息处理 pipeline 的原型——多源采集 + LLM 打分 + 分层输出。把场景换到基金研究（比如某行业的研报合成），方法论完全通用
  • 三层价值：降低研究员信息处理成本 / 提升决策质量 / 让投资决策可追溯
  • 但 AI 落地要避免的坑：不能替代判断，只能辅助——金融场景的信任成本比互联网高得多
Q5. LLM 幻觉在基金场景的代价？怎么控制？
公式：零容忍 → RAG + Source citation + 分级权限 + 人机协同
  • 金融场景零容忍——一个错误的数字可能误导几亿资金的决策
  • 技术手段：
    • RAG（检索外部事实，不让模型瞎编）
    • Source citation（强制引用原文）
    • 拒答机制（没把握不说，比说错好）
  • 产品手段：
    • 人机协同（AI 生成初稿 + 研究员确认）
    • 分级权限（低风险自动 / 高风险必须人工）
    • 审计日志（全量留存，可追溯）
  • 我踩过的坑：AFI 早期 LLM 编造未发生的产品发布，后来加了 source citation 要求
Q6. 如果让你负责"研报智能生成"，第一步做什么？
公式：不急着做产品，先理解现有流程的痛点
  • 第一周：跟研究员跑一遍现有研报生产流程，找 3 个最痛的环节（数据收集 / 图表生成 / 摘要撰写 / 合规检查）
  • 第二周：选一个场景做 MVP（比如"每日行业动态 5 分钟版"——最小验证）
  • 第三周：找 3-5 个真实研究员反馈 → 迭代
  • 原则：不做"大而全的研究员助手"，做"单点解决一个痛"
  • 学 Cursor 的产品哲学：不做新 IDE，在 VS Code 基础上做增强
Q7. 基金数据敏感，AI 部署本地 vs 云端？
公式：混合策略
  • 本地：合规安全，但成本高 + 模型能力受限
  • 云端：成本低 + 模型强，但数据出境 / 合规压力大
  • 混合策略：
    • 非敏感数据（公开研报、新闻）→ 云端处理（用最强模型）
    • 敏感数据（客户信息、持仓、交易）→ 本地小模型
  • 易方达的机会：EFundGPT 的存在说明他们走的就是本地化 + 自研路线——这也是基金行业的普遍选择

C. 技术深度（3 题，JD 要 ML 基础）

Q8. 简单讲一下你理解的 RAG 和 fine-tune 的差异？
  • RAG（检索增强生成）：把外部知识库当 context 喂给 LLM。优点：知识可更新、幻觉可控；缺点：对检索质量依赖高
  • Fine-tune（微调）：用领域数据训练模型本身。优点：领域能力强；缺点：知识固化、更新成本高、需要标注数据
  • 金融场景为什么 RAG 更实用：金融信息时效性强（研报每天更新），RAG 能保证"最新信息"；fine-tune 训完就过时了
  • EFundGPT 可能怎么选：大概率是 RAG + 轻量 fine-tune 组合——用 fine-tune 学金融术语和表达风格，用 RAG 接入实时数据
Q9. 你熟悉哪些机器学习基础概念？
公式：诚实分层——懂什么、做过什么、不会什么
  • 懂基础理论：bias-variance / overfitting / regularization / cross-validation / confusion matrix / precision vs recall
  • 做过具体实践：AFI 里的 LLM 打分 pipeline、prompt engineering、RAG 实现、embedding 召回
  • 不会但能学：深度学习底层（backprop / attention 机制细节 / 训练框架）——我能读 paper 理解原理，但不是训模型的人
  • 重要的是认知：我知道自己的技术边界在哪，也知道该和算法工程师怎么协作
Q10. 你看过的 AI paper 或技术进展？
公式：选 2-3 个你真读过的，一句话讲清 insight

建议选：

  • RAG 相关：Lewis et al. 2020 原论文——为什么 parametric memory + non-parametric memory 组合比任一单独好
  • Agent 框架：ReAct（推理+行动循环）/ Reflexion（自我反思）
  • 长 context：Claude 200K / Gemini 1M 对金融场景（长研报）的意义
  • Instruction tuning：InstructGPT / RLHF 的产品价值
诚实原则：别列你没读过的。只讲你能说出核心 insight 的。说不知道某个进展不丢人，编就丢人。

D. 研究转化 / 协作能力（2 题）

Q11. 研究员可能说"我不需要 AI，我自己研究得更好"。怎么应对？
公式：不对抗 + 找减负点 + AI augment 不 replace
  • 不正面对抗——研究员的专业判断是真的重要
  • 找"可量化减负"的切入点：数据收集 / 图表生成 / 初稿撰写（这些是研究员最不爱做的）
  • 让 AI 做"前置工作"，研究员做"判断工作"——分工而非替代
  • 关键话术："AI 不是来取代你的专业判断，是把你从重复劳动里解放出来，让你有更多时间做高价值判断"
Q12. 如果你的 AI 方案和算法工程师的意见有冲突怎么办？
公式：承认边界 + 数据说话 + 用户反馈是裁判
  • 承认边界：算法工程师在模型层面比我懂——算法实现我听他的
  • 数据说话：产品效果上我们用 A/B 实验验证——让数据做裁判
  • 用户反馈：最终用户（研究员）觉得哪个方案好用，比我们谁对更重要
  • 我的角色：桥梁——把业务痛点翻译给算法，把算法能力翻译给业务
