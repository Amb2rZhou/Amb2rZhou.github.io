二、搜索 PM 基础（出现概率最高的 4 题）

Q3. 召回（recall）和排序（ranking）的区别？协同逻辑？
阶段输入输出关键指标
召回query~1000 候选recall 高 / 延迟低
排序候选集~20-30 展示精排 / 多目标

协同：召回决定"能不能搜到"，排序决定"看得到什么"。召回漏了排序救不回来，排序错了召回再好也白搭。

电商特有：

  • 召回侧平衡 exact match（倒排）和 semantic match（embedding 召回）
  • 排序侧平衡 relevance / CTR / GMV / 多样性 / 商家公平性
加分项："我会特别关注召回和排序两阶段的目标对齐——如果召回模型和排序模型优化的不是同一个北极星，会有 online/offline gap，这是搜索系统常见的坑。"
Q4. 如果让你负责"零结果率"，你会怎么拆解？

答题公式：漏斗拆解 → 归因 → ROI 优先级

可能原因（四层）：

  1. query 侧：拼写错误、生僻词、超长 query、非母语混用
  2. 商品侧：商品库覆盖不全、属性不完整、标题质量低
  3. 召回侧：倒排不命中、向量召回阈值过严
  4. 策略侧：过滤规则过严（价格 / 发货地 / 库存）

优先级判断：

  • 先量化：零结果 query 的频次分布 top 10
  • 再算成本：query 改写（LLM 短期快）→ 商品属性补全（中长期）→ 向量召回升级（基建投入）
关键词：漏斗拆解 / 归因 / ROI / LLM 改写抢救零结果
Q5. 相关性和转化率指标冲突怎么办？（经典搜索 PM 题 🔥）

答题思路：不选边站，讲"场景化加权"

  • 承认冲突：相关性高 ≠ 转化率高（高价品 / 新品 / 小众品）
  • 多目标 LTR：看用户所处阶段
    • 明确需求（搜具体商品）：相关性权重高
    • 探索阶段（搜类目 / 场景词）：多样性 + 转化率权重高
  • 指标不看单点看组合：搜索 CVR + 用户停留 + 次日回搜率 + 跨 session 留存
加分项："有时候相关性低但转化高，反映的是意图澄清不充分——应该做意图识别前置，而不是让排序去 cover。这是产品思维和算法思维的区别。"
Q6. 你会怎么用 LLM 改写 query？落地顺序？

答题思路：按 ROI 优先级，从低风险高收益开始

  1. 拼写纠错（零风险 / 高覆盖）
  2. 同义词扩展（抢救零结果）
  3. 长 query 结构化："适合户外徒步的防水鞋" → {类目:鞋, 场景:户外徒步, 属性:防水}
  4. 多语言翻译 + 改写（东南亚特有）
  5. 多轮意图澄清（最复杂，对话式搜索）
关键词：分级落地 / 零结果抢救先行 / 对话式搜索作为终极形态

三、电商搜索深挖（3 题）

Q7. 用户搜"手机壳"，怎么判断真实意图？

显性 vs 隐性：

  • 显性：想买手机壳
  • 隐性：什么手机型号？什么风格？什么价位？

消歧策略（三层）：

  • 看历史：搜过 / 买过的手机型号
  • 看画像：性别 / 年龄 / 消费能力 / 所在国家
  • 看当下：本次 session 浏览了什么
  • 不确定时：搜索结果页顶部做意图澄清 chip（"您的手机是？iPhone / Samsung / Xiaomi / 其他"）
东南亚特有：不同国家手机型号分布差异大（印尼多 Samsung，菲律宾多 Xiaomi），region-aware 默认排序。
Q8. 东南亚多语言搜索的难点？
  • 语种多：英、泰、越、印尼、马来、繁中、简中，同一市场多语混用
  • 切词难：泰语没空格 / 越南语有声调
  • 商品侧不对齐：商品标题往往只一种语言
  • 跨语言召回：embedding 需要多语言对齐
  • 意图歧义："baju" 在印尼是"衣服"，在其他市场可能是别的含义
你的挂钩："AFI 处理多源信息时碰到过中英混合的 signal normalization 问题——电商搜索里这个问题只会更复杂，因为还涉及商品侧和用户侧的语言匹配。"
Q9. 怎么处理长尾 query？

先量化：长尾 query 频次分布 + 长尾占总搜索的比例

召回端：

  • 向量召回兜底
  • LLM 改写成结构化 query
  • 相关商品推荐（"没找到 X，看看类似的"）

体验端：

  • 搜索建议 / 自动补全 降低长尾 query 产生
  • Empty State 做得更好（意图澄清 / 跳类目搜索 / 相关推荐）

数据端：

  • 长尾 query 单独监控，别被高频覆盖（头部 20% query 可能占流量 80%，改好头部不等于搜索体验好）

四、AFI / intern-clawd 挂钩题（2 题，高概率会问 1 题）

Q10. AFI 项目和电商搜索的关系？

答题公式：三层类比

AFI电商搜索

Q11. intern-clawd 能迁移到 Shopee 搜索的是什么？

不讲技术，讲产品哲学：

  • 核心思想：Claude Code 默认无记忆 → 每次交互重新解释 context；intern-clawd 加 wiki 做长期记忆 → 用户少说废话
  • 迁移到搜索：
    • 用户搜索也有"失忆问题"：每次都是冷启动
    • 长期记忆 = 用户画像 + 搜索历史 + 场景偏好
    • "下次搜'手机壳'直接知道是 iPhone 15 的，你喜欢极简风格，预算 50 马币以下"
  • 产品价值：减少澄清成本 + 提升首屏精准度 + 提高回搜率

五、开放题 / 战略题（3 题，面试官可能挑 1 题压轴）

Q12. AI Agent 会取代搜索吗？

不是 yes/no，分层回答：

  • 短期（1-2 年）：搜索 + Agent 并存。Agent 用于复杂需求 / 长尾 / 决策辅助；搜索用于明确意图 / 高频。
  • 中期（3-5 年）：搜索会"Agent 化"——搜索结果页嵌入 Agent（类似阿里"AI 帮我挑"）。
  • 长期：用户入口可能从搜索框转对话框，但底层召回 + 排序还在，只是界面变了。
电商特有优势：交易闭环让 Agent 天花板高——从推荐到对比到下单完整链路，比通用 Agent 更容易变现。这也是电商是 AI Agent 最佳应用场景的原因。
Q13. 设计一个"场景搜索"（例："户外徒步"返回完整清单）

答题公式：用户 → 价值 → 方案 → MVP → 风险

  1. 目标用户：购物决策困难者（新手 / 节日礼物 / 大件置办）
  2. 核心价值：从"找单品"到"找解决方案"，降低跨品类搜索成本
  3. 技术方案：
    • LLM 做场景分解："户外徒步" → {鞋、背包、水壶、防晒、...}
    • 跨品类召回 + 相互匹配（不能推不兼容的）
    • 结果页：清单式 + 一键加购 + 替换选项
  4. MVP 验证：
    • 先做 3-5 个高频场景（露营 / 新生宿舍 / 宝宝用品）
    • 测 GMV 增量 + 停留时长 + 清单完成率
  5. 风险：
    • 商家公平性（推荐品选择）
    • 清单准确性（LLM hallucination）
    • 体验复杂度（vs 简单搜索框，不能过度设计）
Q14. 东南亚跨境未来 3 年最值得做的 1 个搜索策略产品？

推荐选：多模态搜索（拍立搜 + 语音搜）

理由：

  • 东南亚用户智能机普及率高但打字门槛高（语言多 / 方言 / 拼写不规范）
  • 语音输入避开打字摩擦，拍立搜避开语言描述困难
  • LLM + 多模态模型成本在下降，技术窗口打开
  • 商家侧获益：长尾商品被发现的概率上升

落地路径：

  1. 拍立搜（商品识别）→ 先做家居 / 服饰这种视觉强品类
  2. 语音搜（query 输入）→ 先做高频短 query
  3. 多模态融合（图 + 文描述）→ 长尾澄清

备选（如果面试官对拍立搜不感兴趣）：多轮意图澄清 Agent / 跨境 vs 本地智能分流 / LLM 生成商品描述补全

七、🔗 AI 搜索 × RAG：底层是同一个系统（秘密武器）

核心洞察：RAG（Retrieval-Augmented Generation）的 Retrieval 阶段 = 传统搜索的召回 + 排序。这不是"像"，是同一个东西穿了不同的衣服。

对你的意义：你做 AFI 本质就是在做 RAG 系统，RAG 的 Retrieval 层本质就是搜索策略。所以你其实一直在做"搜索策略 PM"——只不过搜索的对象不是商品，是情报。这让你从"AI 产品 PM"直接挂钩"搜索 PM 的技术栈"，不需要另学，只需要换语境。

7.3 追问的差异（面试官可能会问）

Q：搜索和 RAG 的核心差异在哪？

维度搜索RAG
目标多目标（relevance + CTR + GMV + 多样性 + 公平性）单目标（LLM 拿到对的信息）
用户可见结果直接给用户看 → 体验压力大中间环节 → 用户看不到召回细节
成功指标业务指标（转化 / 留存）下游 LLM 输出质量
商业复杂度高（商家利益 / 广告 / 库存）低（只关心 relevance）
评估难度线上 AB + NDCG 等标准指标LLM 输出质量评估，更难定量
加分回答范式： "RAG 简化了搜索问题，因为它只优化'让 LLM 拿到对的信息'，不用考虑商家公平、多样性、广告位。电商搜索是 RAG 的 hard mode——要在召回+排序两阶段处理多目标冲突，这就是为什么搜索 PM 的工作远不止'相关性'。"

Tier 1 · 全文题（10 题，重点复习 🔥）

Q16. Agent 和传统搜推的关系？会取代搜推吗？

不会取代，分层共存。

  • 搜推是基座（大规模、高并发、低成本、100% 用户）
  • Agent 是增强层（复杂意图、多轮决策、高成本、10-20% 用户）

类比：搜推是超市货架（负责广度），Agent 是导购员（负责深度）。Agent 不应该是独立产品，而是嵌入现有搜推流程的增强能力——搜索结果页加"帮我挑"入口，推荐流加"为什么推这个"交互。

Q17. 搜推系统会被 Agent 取代吗？（深度版）⭐

"搜推不会被取代，但它的角色会发生根本性的变化——从前台产品退到后台引擎。

  • 短期（1-2 年）：搜推和 Agent 分层共存，大多数用户还是用搜索框。
  • 中期（3-5 年）：Agent 成为主要交互入口，但用户跟 Agent 说的每一句话，Agent 在背后都在调搜索接口、推荐接口、比价接口。搜推从"用户直接操作的产品"变成"Agent 调用的工具集"。
  • 长期：真正的竞争点会从"谁的搜推算法好"变成"谁的搜推能力更容易被 Agent 调用、接口更丰富、数据更精准"。"
🇸🇬 Shopee 迁移：Shopee 作为跨境平台，在 "Agent 调用的工具集" 转型中有独特优势——多语言商品数据 + 跨国物流 API + 本地支付通道，这些都是 Agent 时代的稀缺接口。
Q18. Agent 时代搜推的新模式？⭐

"Agent 时代最大的模式转变是推荐权力的转移——从平台侧中心化，转向用户侧去中心化。

  • 传统：你的画像存在平台服务器，推荐算法是黑盒。
  • Agent 时代：用户偏好存在自己设备上（类比 intern-clawd 的 MEMORY.md）、推荐逻辑对用户透明、用户 Agent 主动向平台拉数据。

双刃剑：

  • 机会：率先把商品搜推能力以开放接口形式提供给用户 Agent，成为 Agent 时代最好用的商品数据源。
  • 威胁：用户 Agent 如果能跨平台比价，平台锁定力会显著下降，竞争从用户粘性变成接口竞争。

结论：现在该做的是用 Context 积累建立护城河——用户在平台积累的购买记录、地址、偏好、家庭结构越多，Agent 在这里的表现就越好，迁移成本越高。

Q19. AI 搜索怎么处理"好喝的牛奶"这种主观搜索词？⭐

两层处理 + 冷启动：

  • 群体共识：用评价数据挖掘"好喝"这个标签在用户社区里的映射（评论中高频出现的商品属性）。给一个基线答案。
  • 个人偏好覆盖：用户历史买过全脂 → "好喝"对他大概率是全脂；最近看低脂 → "好喝"可能是低脂。用历史行为把主观词翻译成客观特征。
  • 新用户冷启动：一轮轻量澄清（"您是偏好哪种口味？纯正奶香还是淡一点的？"）。一个问题拿一个信号，不要问多。
  • 结果呈现：给 2-3 个理由各异的选项（"口碑最好"/"您历史偏好"/"今天特惠"）。让用户感受到 AI 在理解他，而不是随机推荐。
🇸🇬 Shopee 迁移：东南亚主观词跨语言场景更难。比如印尼语 "enak"（好吃/好喝/舒服都能用），意图消歧比中文更需要多轮澄清。
Q20. 兴趣推理 vs 行为相似，怎么验证兴趣推理更好？

本质区别：

  • 行为相似："买 A 的人也买了 B"——纯统计关联，上限是历史行为的延伸。
  • 兴趣推理："买奶粉 + 看辅食教程 + 搜 8 个月宝宝 → AI 推断你家有 8 个月宝宝 → 推米粉、磨牙棒"。主动状态推断，不是被动匹配。

AB 测试双指标：

  1. CTR / CVR（基础）：兴趣推理不能比行为相似差
  2. 惊喜度（区别度指标）：推的商品里有多少是用户从未浏览过的品类但最终购买了。惊喜度高说明兴趣推理真的帮用户发现了他没想到的需求——这是行为相似做不到的。

判断：惊喜度高且转化不差 = 兴趣推理真的在工作。

Q21. AI 帮我挑 vs 搜索的本质区别？产品形态？⭐

本质是决策权的分工：

  • 搜索：用户主导决策，AI 只负责找。"搜牛奶"→ 给 200 个，你自己选。
  • 帮我挑：用户让渡决策权，AI 负责找+选。"帮我挑一款牛奶"→ 隐含"你帮我做选择"。需要 AI 理解隐性偏好和当前情境，不是列清单，是给建议。

用户心理：搜索是"我知道自己要什么"；帮我挑是"我懒得想了，AI 比我更懂"。后者对 AI 能力的信任要求更高，推错了代价也更高。

产品形态：

  • 入口：放在搜索结果页顶部（用户搜了但没点购买时，说明在犹豫，这时最适合接手）
  • 对话：最多 2-3 轮。第一轮澄清核心约束（给谁买？预算？），第二轮确认，第三轮给结果。
  • 输出：给 2-3 个选项（用户需要控制感），每个附一句推荐理由。
  • 关键细节：直接可加购，不跳详情页——用户选了就是信任了，再看详情是多此一举。
🇸🇬 Shopee 迁移：东南亚用户打字成本高（多语言切换），"帮我挑" ROI 比国内更高。可以从语音输入切入。
Q22. GMV 突然下降 30%，怎么拆解原因？⭐⭐

四步走：

  1. 确认数据真实性（先不质疑，避免在错误数据上浪费时间）：口径变了？大促后自然回落？数据系统 bug？
  2. 沿公式拆解：GMV = 访问用户数 × 转化率 × 客单价。三个因子先看哪个降了：
    • 用户数降 → 流量问题（外部导流 / SEO 有变化）
    • 转化率降 → 产品或信任问题
    • 客单价降 → 结构性变化（用户在买低价商品）
  3. 维度下钻：地区（某个城市？物流问题？）/ 品类（竞对补贴？）/ 用户群（只有新用户还是老用户流失）/ 时间（持续还是单天异常）
  4. 假设排查：竞对大促？算法调整？政策变化？舆情负面？
回答技巧："我的思路是先验真实性，再沿 GMV 公式逐层拆，同时看是全局还是局部。您希望我从哪个方向深入？" ← 把球抛回去，展示思考框架 + 避免自嗨。
Q23. 怎么设计 AB 测试验证 AI 推荐效果？⭐⭐

五步法：

  1. 定义可证伪假设：不是"AI 推荐更好"，而是"AI 推荐会使 7 天转化率提升 X%，且退货率不上升"。具体、可量化、有方向性。
  2. 分组设计：按用户 ID hash 随机分（不按时间段，避免时间混淆）。先 5% 小流量跑 3 天看有没有灾难性问题，再放大到 50% 跑完整周期。
  3. 指标分三层：
    • 主指标只一个（搜索转化率）——防多重检验
    • 辅助指标看方向（客单价、加购率、对话深度）
    • 护栏指标是红线（退货率、客诉率、页面报错率——变差就暂停）
  4. 实验规范：样本量提前用统计功效算（效果量 5% + 置信度 95% 需要多少用户）。至少跑 7 天覆盖周内周期，避开大促。
  5. 避坑：
    • 新鲜感效应：第 1 周数据可能虚高，看第 2 周稳定性
    • 辛普森悖论：全局好 ≠ 分层好，分用户类型、品类看
    • 多重检验：主指标只定一个，不能后验选最好看的汇报
Q24. 上线前怎么做离线评估？⭐

三层：

  1. 自动化回测指标（广度，跑得快）：
    • Recall@K（推的前 K 个里有没有用户实际买的）
    • NDCG（排序是不是把对的排前面）
    • 覆盖率/多样性（千人千面还是千人一面）
  2. 人工 case review（深度，抓 bad case）：抽 100-200 个代表性场景人工标注。电商特别要关注的 bad case：推了用户过敏商品 / 推了刚买完的 / 推了已下架的 / 推荐和家庭结构不匹配（单身推婴儿用品）。
  3. Agent 特殊评估：纯统计指标评估不了对话质量。用真实对话 log 回放 + LLM-as-judge 批量打分。

诚实的局限：离线保下限（避免灾难），上限靠在线 AB。离线永远有分布偏差——历史数据代表过去的行为，新的 AI 交互会产生历史上没有的模式。

Q25. 搜推广的未来——从面向人到面向 Agent ⭐⭐⭐（主动抛出的亮点题）

核心判断：平台搜推广正在从"面向人"转变为"面向 Agent"。

现在的搜推靠什么影响用户？UI 设计、标题吸引眼球、图片视觉、营销文案——这些是为"人类注意力"设计的。但如果用户的 Agent 替他们看，Agent 不吃标题党，它看的是结构化数据：价格准确吗？评分真实吗？库存有没有？匹配度多少？

平台搜推不会消失，但形态会变——从"把商品展示给人"变成"给 Agent 提供结构化商品数据接口"。广告也一样，从抢用户眼球变成抢 Agent 的优先级排序。

平台该做的三件事：

  1. 隐性 Context 显性化：用户散落在各处的行为数据（买过、搜过、退过什么）转化成结构化档案，让 Agent 能调用。
  2. 用 AI 产品创造新 Context：对话式购物能获取行为数据挖不到的信息（"宝宝乳糖不耐受"这种只有对话才能得到）。
  3. 让 Context 不可迁移：用户在平台积累的购买记录、偏好、家庭信息越多，他们的 Agent 在这里表现越好，迁移到其他平台就会变差。

终局判断：未来的竞争力不是 SKU 多少和物流多快，而是你对用户理解有多深。每个用户在平台上积累的 Context 是独一无二的，这才是真正的护城河。

这题是面向 Agent 时代的亮点题，面试快结束时主动抛出能让面试官记住你。结合你做 intern-clawd 的经验（MEMORY.md = 用户侧 Context）讲得特别有说服力。

Tier 2 · 速览题（8 题，扫核心答题思路）

Q26. 入职第一周做什么？

不急着画原型。Day 1-2 了解现状（跑一遍搜索链路、看数据看板、找"搜了但没买"的 case）→ Day 3-4 定义问题（流失是因为"找不到"还是"选不了"？）→ Day 5 输出 MVP 提案（选一个垂直场景验证，定义 AB 对比指标）。原则：第一周产出不是方案，是问题定义。

Q27. 跨团队协作能力（如果被问"你项目都是一个人做的"）

诚实 + 举具体协作经历：

  • 需求对接：AFI 用户是公司技术高管，根据他们反馈做了几次重要改动（加 benchmark 模块、把 webhook 纯文字改成文档形式）
  • 阻塞破局：内部文档 API 权限申请两周没批，自己绕道用 Lobi 实现了替代方案——典型的 PM 处理依赖阻塞：找替代路径，不等着。
  • 外部协作：参与过跟端到端 coding agent PM 的技术选型讨论和外部访谈。
  • 自己垫 API 费：先跑出效果再争取资源——用最小成本证明价值比申请资源更快。

总结："我的优势是需求定义清晰、技术沟通零成本、自己能验原型，对工程师来说沟通成本低。"

Q28. AI PM 需要会写代码吗？技术背景是优势还是包袱？

净正向，但要主动管理副作用。

三个优势：（1）需求可行性判断不依赖开发；（2）跟工程师沟通零翻译成本；（3）能自己写原型快速验证假设。

诚实包袱：容易陷入技术细节，忘了用户视角。主动习惯：每次回答先说用户场景和业务目标，技术方案放后面。

一句话："技术是看问题的放大镜，但决策的准心要始终对准用户价值，不是对准实现的优雅度。"

Q29. 怎么快速判断 AI 功能是否值得上线？

三维度：

  1. 价值假设是否成立：解决真痛点还是技术秀？10-20 个用户 5 分钟访谈，或看现有数据里的痛点信号。
  2. 成本和可行性：AI 功能成本特殊——训练（一次性）+ 推理（按量线性）+ 维护（持续投入）。按 DAU 估日 token 消耗 × 收益粗比。
  3. 最大风险：AI 风险不是普通 bug 是"表现不稳定导致信任崩塌"。最坏情况下用户损失多少？有 fallback 吗？

判断标准：价值清晰 + 成本可接受 + 风险可控 → 上线；任一不满足 → 原型验证再决定。

Q30. 新鲜感效应消退 vs 产品本身有问题，怎么区分？

看两组数据：

  • 用户时间分群：第一周进入 vs 第二周新进入。第二周新用户数据仍好 = 新鲜感问题；第二周新用户也差 = 产品问题。
  • 使用深度而不只是频次：留下来的用户，对话轮数增加？下单率提升？回搜率？

判断标准：第二周留存用户人均 GMV 没降甚至涨 = 健康消退；全面下滑 = 产品有问题。

Q31. prompt 调优实战方法（从 AFI 来）

7 条实战（4 轮迭代总结）：

  1. 角色定义具体（不是"AI 助手"，而是"服务 25-35 岁精打细算妈妈群体的天猫超市导购"）
  2. 给 few-shot 示例而不是规则
  3. 先解决方向错误再优化细节（调优顺序：意图 → 内容 → 格式）
  4. 约束输出格式（JSON / 固定 markdown）
  5. 分步拆任务（复杂任务拆多 prompt）
  6. 用 bad case 驱动迭代（不是凭感觉改）
  7. 温度/top_p 是最后才调的
Q32. AI 客服/功能上线后满意度降了怎么汇报？

先诊断再汇报（不能拿坏数据直接扔给老板）：

  • 拆根因：按问题类型拆（简单 vs 复杂）+ 按失败模式分类（答错 / 没听懂 / 态度冷）+ 对比基准
  • 做决策：AI 擅长的继续，不擅长的直接转人工
  • 汇报结构：现状 → 根因 → 建议 → 净收益（省 XX 万 + 满意度可恢复）

带着诊断和建议去汇报，而不是带着问题让老板做判断。

Q33. MCP（Model Context Protocol）对电商的影响？

MCP 是什么：Anthropic 推出的开放协议，让 AI Agent 标准化连接外部服务，类比 USB-C。

对电商影响：用户 Agent 可以同时调多平台接口，跨平台搜索、比价、下单。竞争维度从"app 体验"变成"MCP 接口开放度 + 数据丰富度 + API 稳定性"。

威胁：跨平台比价零摩擦，价格竞争加剧；机会：率先开放高质量 MCP 接口，成为 Agent 时代商品数据入口。

总结：MCP 会让平台竞争更激烈，但会者多劳——谁先开放 Agent 接入，谁就先占到 Agent 时代的入口位置。
