Hermes Agent 架构深度研究 & 与 OpenClaw 对比分析

研究主题：Nous Research 开源 AI Agent 框架 Hermes Agent 的架构设计与技术实现
报告类型：技术趋势 / 竞品分析
日期：2026-04-11
核心结论：Hermes Agent 是目前 personal AI agent 赛道中最具"自进化"特色的开源框架。它以 agent loop 为核心、以自动生成 skill 为飞轮，走的是"单用户深度陪伴"路线；而 OpenClaw 以 Gateway 为中枢、以多平台覆盖为护城河，走的是"多设备全场景覆盖"路线。两者在记忆系统、skill 管理、安全模型上的设计取舍，反映了 personal agent 领域两种截然不同的产品哲学。

一、Hermes Agent 是什么

Hermes Agent 是 Nous Research 于 2026 年 2 月发布的开源 AI agent 框架（GitHub，MIT 协议）。截至 2026 年 4 月，GitHub 约 22,000 stars。

官方定位用一句话概括："The agent that grows with you"——住在你服务器上，越用越聪明。

这个定位精准地指出了它和其他 agent 框架的核心差异：不是工具，是伙伴。大多数 agent 框架（包括 OpenClaw）的 skill 需要人工编写和维护，Hermes Agent 能在使用过程中自动生成 skill、自动改进 skill、自动积累对用户的理解。

技术栈：Python（94%），SQLite + FTS5，uv 包管理，pytest 测试框架。

最新版本：v0.7.0（2026.4.3），代号"The Resilience Release"——168 PRs merged，46 issues resolved，689 commits（Release Notes）。

二、核心架构拆解

2.1 架构总览

Hermes Agent 的架构围绕一个核心设计原则：platform-agnostic core——无论从 CLI、Gateway、ACP（Agent Communication Protocol）还是 batch 模式进入，底层共用同一个 AIAgent 类（Architecture Docs）。

2.2 ReAct Loop

核心循环实现了经典的 ReAct（Reasoning and Acting）模式：

User input → process_input()
  → AIAgent.run_conversation()
    → prompt_builder.build_system_prompt()
    → runtime_provider.resolve_runtime_provider()
    → API call (chat_completions / codex_responses / anthropic)
    → tool_calls? → model_tools.handle_function_call()
    → loop until done

三个关键设计点：

  1. 三种 API 模式：支持 OpenAI chat completions、Codex responses、Anthropic 原生三种协议，覆盖 18+ provider
  2. 可中断：loop 支持 mid-flight interruption，用户可以随时打断
  3. Prompt 稳定性：system prompt 在 session 内不可变（frozen snapshot），只有用户主动操作才会触发重建——这是为了最大化 prefix cache 命中率

2.3 Prompt Assembly Pipeline

Prompt 组装是 Hermes Agent 最精细的子系统之一。prompt_builder.py 按固定顺序拼接以下内容：

顺序来源说明
1SOUL.md人格定义（类似 OpenClaw 的 SOUL.md）
2MEMORY.mdAgent 的笔记：环境信息、惯例、学到的东西
3USER.md用户画像：偏好、沟通风格、期望
4Skills按需加载的知识文档
5Context files.hermes.md, AGENTS.md 等项目级上下文
6Tool-use guidance工具使用说明
7Model-specific针对不同模型的特化指令

当 context 超过模型限制时，context_compressor.py 执行有损压缩（lossy summarization）。对 Anthropic 模型，prompt_caching.py 自动插入 cache breakpoints 实现 prefix caching。

三、记忆系统

Hermes Agent 的记忆系统是它"越用越懂你"的核心支撑。分三层（Memory Docs）：

3.1 即时记忆：MEMORY.md + USER.md

文件容量内容
MEMORY.md~2,200 chars / ~800 tokensAgent 的个人笔记——环境信息、惯例、学到的东西
USER.md~1,375 chars / ~500 tokens用户画像——偏好、沟通风格、期望
关键设计：Frozen Snapshot 模式。两个文件在 session 开始时做一次快照注入 system prompt，session 内不再变化。修改通过 memory tool 立即写盘，但下次 session 才生效。目的是保证 prefix cache 命中——如果 system prompt 每次工具调用后都变，cache 全废。

容量管理：80% 利用率触发整合提示，防止无限膨胀。

安全：记忆条目写入前做注入和渗透模式扫描，重复条目自动拒绝。

3.2 历史记忆：SQLite + FTS5

所有 CLI 和 Gateway session 存储在 ~/.hermes/state.db（SQLite），FTS5 全文索引支持跨 session 搜索。session_search 工具结合 Gemini Flash 做摘要，实现"几个月前聊过的事也能找回来"。

  • Lineage tracking：跨压缩的父子关系追踪
  • Per-platform isolation：不同消息平台的 session 相互隔离
  • Atomic writes：并发写入安全
  • Profile scoping：hermes -p <name> 每个 profile 独立 HERMES_HOME

四、Skills 系统——核心差异化

Skills 是 Hermes Agent 最具特色的子系统，也是它和其他 agent 框架拉开差距的地方（Skills Docs）。

4.1 Skill 格式

遵循 agentskills.io 开放标准。SKILL.md 结构：

---
name: skill-name
description: "..."
version: "1.0"
platforms: [macos, linux]
metadata:
  hermes:
    tags: [...]
    category: "..."
    fallback_for_toolsets: [...]
    requires_toolsets: [...]
---

# When to Use
# Quick Reference
# Procedure
# Pitfalls
# Verification

4.2 渐进式加载（Progressive Disclosure）

三级加载最小化 token 消耗：

Level操作Token 成本
L0skills_list()~3k tokens（只返回 metadata）
L1skill_view(name)加载完整内容
L2skill_view(name, path)访问特定引用文件

4.3 自动生成 + 自我改进

核心飞轮：完成复杂任务（5+ tool calls）→ 自动生成 SKILL.md → 下次类似任务自动加载 → 使用中 patch 改进 → 越用越精准

触发条件：

  • 完成复杂任务（5+ 次工具调用）
  • 从错误中恢复
  • 发现新的工作流模式
  • 用户纠正后的经验总结

skill_manage 工具操作： create / patch / edit / write_file / remove_file

4.4 条件激活（Fallback Skills）

Skill 可以声明在特定工具不可用时自动出现。例如：DuckDuckGo skill 只在 web toolset 缺少 API key 时自动出现——作为降级方案。

六、多平台网关

Gateway 是一个长驻进程，通过 14 个平台适配器复用统一的 session 路由：Telegram、Discord、Slack、WhatsApp、Signal、Matrix、Mattermost、Email、SMS、DingTalk、Feishu、WeChat、BlueBubbles、Webhook。

跨平台会话连续：可以在 CLI 上开始一个任务，Telegram 上收到通知，Discord 上继续——不丢上下文（The New Stack）。

七、其他能力

7.1 Cron 调度

内置 cron 调度器，支持自然语言定义定时任务。任务结果可投递到任意已连接平台。

7.2 Subagent 委派

支持生成隔离的子 agent，并行处理多个任务流。每个 subagent 有独立的 terminal 和 Python RPC 接口。

7.3 浏览器自动化

v0.7.0 新增 Camofox 反检测浏览器后端，支持持久化 session 和 VNC 调试。

7.4 安全机制

  • Secret 渗透扫描：扫描浏览器 URL 和 LLM 响应中的 URL 编码/base64/注入模式
  • 凭证目录保护：.docker、.azure、.config/gh 等目录自动屏蔽
  • 沙箱输出脱敏：execute_code 的输出做敏感数据 redaction
  • 零遥测：不收集用户数据

7.5 OpenClaw 迁移

内置 hermes claw migrate 命令，一键导入 OpenClaw 的 SOUL.md、记忆、Skills、Allowlist、平台配置、API 凭证。支持 --dry-run 预览。

8.1 架构哲学：Agent Loop vs Gateway Platform

Hermes AgentOpenClaw
核心引擎AIAgent（agent loop 本身）Gateway（WebSocket 控制面）
设计中心Agent 的执行循环是一切的起点消息路由 + 权限是一切的起点

九、关键洞察

9.1 "自进化" vs "手动策展"是 personal agent 的核心路线之争。
Hermes 赌的是：模型足够强之后，agent 可以自己积累知识。OpenClaw 赌的是：平台覆盖和用户习惯才是护城河。目前看，两条路都走得通，但 Hermes 的路线如果跑通，长期护城河更深——因为每个用户的 agent 都是独一无二的。
9.2 Frozen Snapshot 是一个值得借鉴的工程模式。
把记忆在 session 开始时冻结、session 内不变，换取 prefix cache 稳定命中。这个 trade-off 在 API 成本敏感的场景下非常有价值。
9.3 渐进式 Skill 加载值得参考。
L0（metadata）→ L1（全文）→ L2（引用文件）的三级加载，可以有效控制 prompt 膨胀。对于 skill 数量多的系统来说，这比一次性全部加载高效得多。
9.4 Hermes 的 OpenClaw 迁移命令是一步好棋。
内置 hermes claw migrate 直接降低了竞品用户的迁移成本。OpenClaw 的安全事件（CVE-2026-25253）可能加速了一部分用户向 Hermes 的迁移。
